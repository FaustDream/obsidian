# 悲观锁和乐观锁使用场景

## 悲观锁和乐观锁的使用场景

### 悲观锁使用场景

**适用情况：**

- **高并发写操作**：当系统中写操作频繁，冲突概率高时
- **数据一致性要求极高**：如银行转账、库存扣减等关键业务
- **冲突成本高**：重试成本很高，希望一次性成功完成操作
- **短时间持有锁**：操作能够快速完成，避免长时间阻塞

**典型应用场景：**

```java
// 1. 数据库行锁 - 库存扣减
@Transactional
public void decreaseStock(Long productId, int quantity) {
    // SELECT ... FOR UPDATE 获取行锁
    Product product = productMapper.selectByIdForUpdate(productId);
    if (product.getStock() >= quantity) {
        product.setStock(product.getStock() - quantity);
        productMapper.updateById(product);
    }
}

// 2. 分布式锁 - 防止重复处理
public void processOrder(String orderId) {
    String lockKey = "order:" + orderId;
    if (redisLock.tryLock(lockKey, 30, TimeUnit.SECONDS)) {
        try {
            // 处理订单逻辑
            doProcessOrder(orderId);
        } finally {
            redisLock.unlock(lockKey);
        }
    }
}
```

### 乐观锁使用场景

**适用情况：**

- **读多写少**：系统以查询为主，更新操作相对较少
- **冲突概率低**：多个用户同时修改同一数据的可能性较小
- **可接受重试**：冲突时重试成本不高，用户体验可接受
- **高并发读取**：不会因为锁而影响读取性能

**典型应用场景：**

```java
// 1. 版本号机制 - 用户信息更新
@Entity
public class User {
    @Version
    private Long version;
    // 其他字段...
}

public boolean updateUser(User user) {
    try {
        // JPA会自动处理version字段
        userRepository.save(user);
        return true;
    } catch (OptimisticLockException e) {
        // 版本冲突，提示用户重新获取数据
        return false;
    }
}

// 2. CAS操作 - 计数器
public class Counter {
    private AtomicInteger count = new AtomicInteger(0);
    
    public int increment() {
        return count.incrementAndGet(); // 内部使用CAS
    }
}
```

### 场景对比分析

|维度|悲观锁|乐观锁|
|---|---|---|
|**并发性能**|写操作串行化，性能较低|读操作无锁，性能较高|
|**适用场景**|写多读少，冲突频繁|读多写少，冲突较少|
|**数据一致性**|强一致性保证|最终一致性|
|**系统复杂度**|需要处理死锁、超时|需要处理重试逻辑|

### 混合使用策略

在实际项目中，往往需要根据具体业务场景灵活选择：

```java
// 根据业务特点选择锁策略
public class OrderService {
    
    // 高频写操作使用悲观锁
    public void payOrder(Long orderId) {
        Order order = orderMapper.selectByIdForUpdate(orderId);
        if (order.getStatus() == OrderStatus.PENDING) {
            order.setStatus(OrderStatus.PAID);
            orderMapper.updateById(order);
        }
    }
    
    // 低频更新使用乐观锁
    public boolean updateOrderRemark(Long orderId, String remark, Long version) {
        Order order = new Order();
        order.setId(orderId);
        order.setRemark(remark);
        order.setVersion(version);
        
        return orderMapper.updateWithVersion(order) > 0;
    }
}
```

### 扩展知识点

**1. 分布式场景下的锁选择**

- Redis分布式锁：适用于悲观锁场景
- 数据库版本号：适用于乐观锁场景
- Zookeeper：适用于强一致性要求的悲观锁

**2. 性能优化考虑**

- 悲观锁：缩短锁持有时间，避免长事务
- 乐观锁：合理设计重试机制，避免无限重试

**3. 实际应用建议**

- 关键业务（支付、库存）：优先考虑悲观锁
- 用户配置更新：优先考虑乐观锁
- 高并发计数：使用原子操作或CAS

选择锁策略的核心原则是：**根据业务特点权衡一致性、性能和复杂度**，没有绝对的好坏，只有最适合的场景。
