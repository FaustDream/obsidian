# 乐观锁和悲观锁的区别

## 乐观锁和悲观锁的区别

### 核心思想差异

**悲观锁（Pessimistic Locking）**
> 悲观地认为其它线程会修改数据，所以在访问数据前就获取锁，确保独占访问

- 假设并发冲突一定会发生
- 在访问数据前就获取锁，确保独占访问
- "悲观"地认为其他线程会修改数据

**乐观锁（Optimistic Locking）**
> 乐观地认为大多数情况下不会有冲突，在更新时才检查数据是否被修改

- 假设并发冲突很少发生
- 不提前加锁，在更新时才检查数据是否被修改
- "乐观"地认为大多数情况下不会有冲突

### 实现方式

**悲观锁实现：**

```java
// 1. synchronized关键字
public synchronized void updateAccount(Account account) {
    // 修改账户余额
    account.setBalance(account.getBalance() + 100);
}

// 2. ReentrantLock
private final ReentrantLock lock = new ReentrantLock();
public void updateAccount(Account account) {
    lock.lock();
    try {
        account.setBalance(account.getBalance() + 100);
    } finally {
        lock.unlock();
    }
}

// 3. 数据库行锁
SELECT * FROM account WHERE id = 1 FOR UPDATE;
```

**乐观锁实现：**

```java
// 1. CAS (Compare And Swap)
private AtomicInteger balance = new AtomicInteger(1000);
public void updateBalance(int amount) {
    int current;
    int newValue;
    do {
        current = balance.get();
        newValue = current + amount;
    } while (!balance.compareAndSet(current, newValue));
}

// 2. 版本号机制
@Entity
public class Account {
    @Version
    private Long version;
    private Integer balance;
    // getters and setters
}

// 更新时检查版本号
UPDATE account SET balance = ?, version = version + 1 
WHERE id = ? AND version = ?;
```

### 性能特点

**悲观锁：**

- 适用于写操作频繁的场景
- 避免了重试成本
- 但会造成线程阻塞，降低并发性能

**乐观锁：**

- 适用于读操作频繁的场景
- 无锁等待，并发性能高
- 但在高冲突场景下重试成本高

### 使用场景

**悲观锁适用场景：**

- 数据更新频繁
- 冲突概率高
- 对一致性要求严格
- 例如：银行转账、库存扣减

**乐观锁适用场景：**

- 读操作远多于写操作
- 冲突概率低
- 对响应时间要求高
- 例如：商品浏览、文章阅读统计

### 实际应用示例

**电商库存扣减场景：**

```java
// 悲观锁方案
@Transactional
public void decreaseStock(Long productId, int quantity) {
    // 先锁定库存记录
    Product product = productMapper.selectForUpdate(productId);
    if (product.getStock() >= quantity) {
        product.setStock(product.getStock() - quantity);
        productMapper.update(product);
    } else {
        throw new InsufficientStockException();
    }
}

// 乐观锁方案
@Transactional
public void decreaseStock(Long productId, int quantity) {
    Product product = productMapper.selectById(productId);
    if (product.getStock() >= quantity) {
        int updateCount = productMapper.updateStockWithVersion(
            productId, quantity, product.getVersion());
        if (updateCount == 0) {
            throw new OptimisticLockException("库存更新失败，请重试");
        }
    } else {
        throw new InsufficientStockException();
    }
}
```

### 相关知识扩展

**1. ABA问题**

- 乐观锁可能遇到ABA问题（值从A变为B再变回A）
- 解决方案：使用版本号或时间戳

**2. 自旋锁**

- CAS失败时的重试机制
- 避免线程切换开销，但可能浪费CPU

**3. 数据库隔离级别**

- READ COMMITTED：避免脏读
- REPEATABLE READ：避免不可重复读
- SERIALIZABLE：避免幻读

这两种锁机制在实际开发中往往结合使用，根据具体业务场景选择最适合的方案。
