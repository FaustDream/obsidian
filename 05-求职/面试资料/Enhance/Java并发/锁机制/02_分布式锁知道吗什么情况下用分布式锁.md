# 分布式锁知道吗？什么情况下用分布式锁

分布式锁是在分布式系统中用来确保多个节点对共享资源进行互斥访问的同步机制。

## 什么是分布式锁

分布式锁本质上是一种跨进程、跨服务器的锁机制，它可以保证在分布式环境下，同一时刻只有一个客户端能够持有锁并访问共享资源。

## 什么情况下使用分布式锁

### 1. 防止重复执行

- **定时任务**：多个实例部署时，防止定时任务重复执行
- **幂等性控制**：防止用户重复提交订单、重复扣款等

### 2. 共享资源访问控制

- **库存扣减**：高并发下防止超卖
- **唯一性约束**：生成全局唯一ID、用户名注册等
- **缓存重建**：防止缓存击穿时多个线程同时重建缓存

### 3. 分布式事务协调

- **资源预占**：在分布式事务中预先锁定资源
- **状态同步**：确保分布式系统状态的一致性

## 常见的分布式锁实现方案

### 1. 基于Redis的分布式锁

```java
// 使用SET命令的原子性
String lockKey = "lock:order:" + orderId;
String lockValue = UUID.randomUUID().toString();
String result = jedis.set(lockKey, lockValue, "PX", 30000, "NX");

if ("OK".equals(result)) {
    try {
        // 执行业务逻辑
        processOrder();
    } finally {
        // 使用Lua脚本确保原子性释放锁
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end";
        jedis.eval(script, Collections.singletonList(lockKey), Collections.singletonList(lockValue));
    }
}
```

### 2. 基于ZooKeeper的分布式锁

```java
// 创建临时顺序节点
String lockPath = "/locks/order_" + orderId;
String currentPath = zookeeper.create(lockPath + "/", new byte[0], 
    ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL_SEQUENTIAL);

// 获取所有子节点并排序
List<String> children = zookeeper.getChildren("/locks", false);
Collections.sort(children);

// 判断是否是最小节点
if (currentPath.endsWith(children.get(0))) {
    // 获得锁，执行业务逻辑
    processOrder();
} else {
    // 监听前一个节点
    String watchPath = children.get(children.indexOf(currentPath.substring(currentPath.lastIndexOf("/") + 1)) - 1);

}
```

### 3. 基于数据库的分布式锁

```java
// 使用数据库唯一索引
try {
    String sql = "INSERT INTO distributed_lock (lock_key, lock_value, expire_time) VALUES (?, ?, ?)";
    int result = jdbcTemplate.update(sql, lockKey, lockValue, System.currentTimeMillis() + 30000);
    if (result > 0) {
        // 获得锁
        processOrder();
    }
} catch (DuplicateKeyException e) {
    // 锁已被占用
    return false;
} finally {
    // 释放锁
    jdbcTemplate.update("DELETE FROM distributed_lock WHERE lock_key = ? AND lock_value = ?", lockKey, lockValue);
}
```

## 各种方案的对比

|方案|优点|缺点|适用场景|
|---|---|---|---|
|Redis|性能高、实现简单|可能存在锁失效问题|高并发、对一致性要求不是特别严格|
|ZooKeeper|强一致性、可靠性高|性能相对较低、复杂度高|对一致性要求很高的场景|
|数据库|简单易懂、事务支持好|性能最差、存在单点故障|并发量不高、已有数据库的场景|

## 分布式锁的关键问题

### 1. 锁超时问题

业务执行时间超过锁的过期时间，可能导致锁被误释放。 **解决方案**：实现锁续期机制（看门狗机制）

### 2. 锁误释放问题

A进程的锁过期后，B进程获得锁，A进程完成业务后误释放了B的锁。 **解决方案**：释放锁时验证锁的持有者身份

### 3. 不可重入问题

同一个线程无法重复获取同一把锁。 **解决方案**：记录锁的持有者信息和重入次数

### 4. 死锁问题

进程获得锁后异常退出，锁无法释放。 **解决方案**：设置锁过期时间、使用临时节点

## 实际应用示例

以电商系统的库存扣减为例：

```java
@Service
public class InventoryService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    public boolean deductInventory(Long productId, int quantity) {
        String lockKey = "inventory:lock:" + productId;
        String lockValue = UUID.randomUUID().toString();
        
        try {
            // 获取分布式锁
            Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, lockValue, Duration.ofSeconds(30));
            
            if (acquired) {
                // 检查库存
                int currentStock = getCurrentStock(productId);
                if (currentStock >= quantity) {
                    // 扣减库存
                    updateStock(productId, currentStock - quantity);
                    return true;
                }
                return false;
            }
        } finally {
            // 释放锁（使用Lua脚本保证原子性）
            releaseLock(lockKey, lockValue);
        }
        
        return false;
    }
}
```

分布式锁是分布式系统中保证数据一致性的重要工具，但使用时需要仔细考虑性能、可靠性和实现复杂度的平衡。
