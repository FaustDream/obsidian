# 线程池解决什么问题

线程池主要解决以下几个核心问题：

## 1. 线程创建和销毁的性能开销

**问题**：频繁创建和销毁线程会消耗大量系统资源

- 每次创建线程需要分配内存空间、初始化线程栈
- 销毁线程需要回收资源，触发GC
- 这些操作在高并发场景下会成为性能瓶颈

**解决方案**：线程池预先创建一定数量的线程并复用，避免频繁的创建销毁操作。

```java
// 不使用线程池 - 每次都创建新线程
for (int i = 0; i < 1000; i++) {
    new Thread(() -> {
        // 执行任务
    }).start(); // 创建1000个线程，开销巨大
}

// 使用线程池 - 复用线程
ExecutorService executor = Executors.newFixedThreadPool(10);
for (int i = 0; i < 1000; i++) {
    executor.submit(() -> {
        // 执行任务
    }); // 只使用10个线程处理1000个任务
}
```

## 2. 资源管控问题

**问题**：无限制创建线程可能导致系统资源耗尽

- 每个线程占用约1MB内存（默认栈大小）
- 过多线程会导致频繁的上下文切换，降低CPU效率
- 可能出现OOM（内存溢出）

**解决方案**：线程池通过限制线程数量来控制资源消耗。

## 3. 任务管理和调度问题

**问题**：直接使用Thread难以管理大量异步任务

- 无法有效控制任务的执行顺序
- 难以获取任务执行结果
- 缺乏任务队列机制

**解决方案**：线程池提供了完整的任务管理机制。

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5,                      // 核心线程数
    10,                     // 最大线程数
    60L,                    // 空闲线程存活时间
    TimeUnit.SECONDS,       // 时间单位
    new LinkedBlockingQueue<>(100), // 任务队列
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
);
```

## 4. 系统稳定性问题

**问题**：线程管理不当可能导致系统不稳定

- 线程泄露（创建后未正确关闭）
- 死锁风险增加
- 异常处理困难

**解决方案**：线程池提供统一的生命周期管理和异常处理机制。

## 延伸知识点

### 线程池的核心参数理解

- **corePoolSize**：核心线程数，即使空闲也会保留
- **maximumPoolSize**：最大线程数，超过core时才创建
- **keepAliveTime**：非核心线程的空闲存活时间
- **workQueue**：任务队列，常见有ArrayBlockingQueue、LinkedBlockingQueue
- **rejectedExecutionHandler**：拒绝策略，如AbortPolicy、CallerRunsPolicy

### 常见线程池类型

1. **FixedThreadPool**：固定大小线程池
2. **CachedThreadPool**：缓存线程池，适合大量短任务
3. **SingleThreadExecutor**：单线程池，保证任务顺序执行
4. **ScheduledThreadPool**：定时任务线程池

### 最佳实践

- 根据任务类型选择合适的线程池大小（CPU密集型 vs IO密集型）
- 合理设置队列大小，避免内存溢出
- 选择合适的拒绝策略
- 及时关闭线程池，避免资源泄露

线程池本质上是一种**对象池模式**的应用，通过复用和管控实现了性能优化和资源管理的平衡。
