# 线程池的核心线程数的设置依据是什么

线程池核心线程数的设置主要依据以下几个关键因素：

## 1. 任务类型分析

**CPU密集型任务**：核心线程数通常设置为 `CPU核心数 + 1`

- 因为CPU密集型任务主要消耗CPU资源，过多线程会导致频繁的上下文切换
- +1是为了当某个线程因为页缺失或其他原因暂停时，额外的线程可以充分利用CPU

**IO密集型任务**：核心线程数可以设置为 `CPU核心数 × 2` 或更多

- IO操作会阻塞线程，此时CPU处于空闲状态
- 可以用公式：`核心线程数 = CPU核心数 × (1 + IO等待时间/CPU计算时间)`

## 2. 系统资源考量

```java
// 获取CPU核心数
int cpuCores = Runtime.getRuntime().availableProcessors();

// CPU密集型
int corePoolSize = cpuCores + 1;

// IO密集型
int corePoolSize = cpuCores * 2;
```

## 3. 业务特性分析

- **任务执行时间**：短任务可以设置较少核心线程，长任务需要更多
- **任务优先级**：高优先级任务可能需要预留更多核心线程
- **任务到达频率**：高频任务建议设置较多核心线程保持常驻

## 4. 内存限制

每个线程大约占用1MB的栈空间，需要考虑：

```java
// 估算内存使用
long memoryPerThread = 1024 * 1024; // 1MB
long availableMemory = Runtime.getRuntime().maxMemory();
int maxThreadsByMemory = (int)(availableMemory / memoryPerThread / 4); // 预留75%给其他用途
```

## 5. 实际应用场景举例

**Web服务器场景**：

```java
// Tomcat默认配置参考
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,  // 核心线程数：处理常规请求
    200, // 最大线程数：应对突发流量
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100)
);
```

**数据处理场景**：

```java
// 大数据批处理
int corePoolSize = Math.max(2, cpuCores - 1); // 保留一个核心给系统
```

## 6. 动态调整策略

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(...);

// 运行时监控和调整
public void adjustCorePoolSize() {
    int activeCount = executor.getActiveCount();
    int queueSize = executor.getQueue().size();
    
    if (queueSize > threshold && activeCount == executor.getCorePoolSize()) {
        executor.setCorePoolSize(executor.getCorePoolSize() + 1);
    }
}
```

## 7. 监控指标

设置核心线程数后，需要监控：

- 线程池活跃线程数
- 任务队列长度
- 任务平均执行时间
- 系统CPU使用率
- 内存使用情况

## 最佳实践建议

1. **先估算再测试**：根据理论公式估算初始值，然后通过压力测试验证
2. **预留安全边界**：不要将资源用满，预留20-30%的缓冲
3. **支持动态调整**：使用`ThreadPoolExecutor`的`setCorePoolSize()`方法支持运行时调整
4. **建立监控机制**：通过JMX或自定义监控收集线程池运行数据

合理的核心线程数设置能够在保证系统响应性的同时，避免资源浪费和系统过载。
