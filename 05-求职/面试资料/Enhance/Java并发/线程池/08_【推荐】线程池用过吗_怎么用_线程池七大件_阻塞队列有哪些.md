# 线程池用过吗，怎么用，线程池七大件，阻塞队列有哪些

我来详细回答线程池相关的问题。

## 线程池使用经验

线程池在实际开发中经常使用，主要用于管理和复用线程，避免频繁创建和销毁线程的开销。

### 基本使用方式

```java
// 1. 通过Executors工具类创建（不推荐生产环境）
ExecutorService executor = Executors.newFixedThreadPool(5);

// 2. 手动创建ThreadPoolExecutor（推荐）
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2,                      // 核心线程数
    5,                      // 最大线程数
    60L,                    // 空闲线程存活时间
    TimeUnit.SECONDS,       // 时间单位
    new LinkedBlockingQueue<>(100),  // 阻塞队列
    Executors.defaultThreadFactory(), // 线程工厂
    new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
);

// 提交任务
executor.submit(() -> {
    System.out.println("任务执行中...");
});
```

## 线程池七大参数

ThreadPoolExecutor构造函数的七个核心参数：

1. **corePoolSize（核心线程数）**
    
    - 线程池中始终保持的线程数量
    - 即使线程空闲也不会被回收
2. **maximumPoolSize（最大线程数）**
    
    - 线程池中允许的最大线程数量
    - 当队列满了且当前线程数小于最大线程数时，会创建新线程
3. **keepAliveTime（空闲线程存活时间）**
    
    - 非核心线程空闲时的存活时间
    - 超过这个时间的空闲线程会被回收
4. **unit（时间单位）**
    
    - keepAliveTime的时间单位
    - 如TimeUnit.SECONDS、TimeUnit.MINUTES等
5. **workQueue（阻塞队列）**
    
    - 存储等待执行任务的队列
    - 当核心线程都在忙碌时，新任务会放入队列
6. **threadFactory（线程工厂）**
    
    - 创建线程的工厂
    - 可以自定义线程名称、优先级等
7. **handler（拒绝策略）**
    
    - 当队列满且线程数达到最大值时的处理策略

## 阻塞队列类型

### 1. ArrayBlockingQueue

- **特点**: 基于数组的有界阻塞队列
- **容量**: 固定大小，创建时必须指定
- **使用场景**: 需要严格控制队列大小的场景

```java
new ArrayBlockingQueue<>(100) // 容量为100
```

### 2. LinkedBlockingQueue

- **特点**: 基于链表的阻塞队列
- **容量**: 可以有界也可以无界（默认Integer.MAX_VALUE）
- **使用场景**: Executors.newFixedThreadPool() 使用此队列

```java
new LinkedBlockingQueue<>()      // 无界队列
new LinkedBlockingQueue<>(100)   // 有界队列，容量100
```

### 3. SynchronousQueue

- **特点**: 不存储元素的阻塞队列
- **容量**: 0，每个put操作必须等待take操作
- **使用场景**: Executors.newCachedThreadPool() 使用此队列

### 4. PriorityBlockingQueue

- **特点**: 支持优先级的无界阻塞队列
- **排序**: 元素按优先级排序
- **使用场景**: 需要按优先级处理任务的场景

### 5. DelayQueue

- **特点**: 支持延时获取元素的无界阻塞队列
- **使用场景**: 延时任务调度

### 6. LinkedTransferQueue

- **特点**: 基于链表的无界阻塞队列
- **特性**: 支持transfer操作，比SynchronousQueue性能更好

## 线程池执行流程

1. 提交任务时，如果核心线程数未满，创建核心线程执行任务
2. 核心线程满了，任务放入阻塞队列等待
3. 队列满了，创建非核心线程执行任务（不超过最大线程数）
4. 线程数达到最大值且队列满了，执行拒绝策略

## 拒绝策略

1. **AbortPolicy**: 抛出RejectedExecutionException异常（默认）
2. **CallerRunsPolicy**: 由调用线程执行任务
3. **DiscardPolicy**: 静默丢弃任务
4. **DiscardOldestPolicy**: 丢弃队列中最老的任务

## 实际应用建议

- 生产环境避免使用Executors工具类，手动创建ThreadPoolExecutor
- 根据任务类型选择合适的阻塞队列
- CPU密集型任务：线程数 = CPU核心数 + 1
- IO密集型任务：线程数 = CPU核心数 × 2

这样既保证了线程池的高效使用，又避免了常见的线程池配置问题。
