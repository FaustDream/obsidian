# 死锁的产生条件？预防措施？

死锁是多线程编程中的一个经典问题。让我为你详细解释死锁的产生条件和预防措施。

## 死锁的四个必要条件

死锁的产生需要同时满足以下四个条件（这四个条件缺一不可）：

### 1. 互斥条件（Mutual Exclusion）

资源在同一时刻只能被一个线程使用。比如一个对象的synchronized方法，同时只能有一个线程执行。

### 2. 请求与保持条件（Hold and Wait）

线程已经持有至少一个资源，同时又在请求其他被别的线程占用的资源。

### 3. 不可剥夺条件（No Preemption）

资源不能被强制性地从持有它的线程中剥夺，只能由持有资源的线程主动释放。

### 4. 循环等待条件（Circular Wait）

存在一个线程的循环链，每个线程都在等待下一个线程所持有的资源。

## 经典的死锁示例

```java
public class DeadlockExample {
    private static Object lock1 = new Object();
    private static Object lock2 = new Object();
    
    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lock1) {
                System.out.println("Thread 1: 持有lock1");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lock2) {
                    System.out.println("Thread 1: 持有lock2");
                }
            }
        });
        
        Thread t2 = new Thread(() -> {
            synchronized (lock2) {
                System.out.println("Thread 2: 持有lock2");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lock1) {
                    System.out.println("Thread 2: 持有lock1");
                }
            }
        });
        
        t1.start();
        t2.start();
    }
}
```

## 死锁的预防措施

### 1. 破坏互斥条件

这通常不现实，因为很多资源天然具有互斥性。

### 2. 破坏请求与保持条件

**一次性申请所有资源**：

```java
// 使用tryLock避免无限等待
if (lock1.tryLock()) {
    try {
        if (lock2.tryLock()) {
            try {
                // 执行业务逻辑
            } finally {
                lock2.unlock();
            }
        }
    } finally {
        lock1.unlock();
    }
}
```

### 3. 破坏不可剥夺条件

**使用带超时的锁**：

```java
if (lock1.tryLock(1000, TimeUnit.MILLISECONDS)) {
    try {
        if (lock2.tryLock(1000, TimeUnit.MILLISECONDS)) {
            try {
                // 执行业务逻辑
            } finally {
                lock2.unlock();
            }
        }
    } finally {
        lock1.unlock();
    }
}
```

### 4. 破坏循环等待条件

**按顺序获取锁**：

```java
// 为所有锁分配一个唯一的序号，按序号顺序获取
private static final Object lock1 = new Object(); // 序号1
private static final Object lock2 = new Object(); // 序号2

// 所有线程都按照 lock1 -> lock2 的顺序获取锁
synchronized (lock1) {
    synchronized (lock2) {
        // 业务逻辑
    }
}
```

## 实际开发中的最佳实践

### 1. 使用java.util.concurrent包

```java
// 使用ReentrantLock替代synchronized
ReentrantLock lock1 = new ReentrantLock();
ReentrantLock lock2 = new ReentrantLock();

// 支持超时和中断
if (lock1.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (lock2.tryLock(1, TimeUnit.SECONDS)) {
            try {
                // 业务逻辑
            } finally {
                lock2.unlock();
            }
        }
    } finally {
        lock1.unlock();
    }
}
```

### 2. 死锁检测工具

- **JVM参数**：`-XX:+PrintGCDetails -XX:+PrintConcurrentLocks`
- **jstack命令**：`jstack <pid>` 查看线程堆栈
- **JConsole/VisualVM**：图形化监控工具

### 3. 代码设计原则

- 尽量减少锁的使用
- 缩小锁的作用域
- 使用无锁编程（如CAS操作）
- 避免嵌套锁

## 相关知识延伸

### 活锁（Livelock）

线程没有被阻塞，但由于某些条件没有满足，导致线程持续重试而无法继续执行。

### 饥饿（Starvation）

某些线程因为优先级低或其他原因，长期无法获得所需资源。

### 银行家算法

操作系统中用于避免死锁的经典算法，通过预先分析资源分配的安全性来避免死锁。

理解死锁不仅有助于编写更安全的多线程代码，也是面试中经常考查的重点内容。在实际开发中，推荐使用`java.util.concurrent`包中的高级并发工具，它们已经帮我们处理了很多并发问题。
