# 项目中那个环节让你印象最深刻？为什么？

这是一个很好的面试问题，让我分享一个印象深刻的项目经历：

## 最深刻的项目环节：分布式事务处理的设计与实现

### 具体场景

在一个电商系统中，我负责设计订单服务，涉及到库存扣减、订单创建、支付处理、积分增加等多个服务的协调。这个环节让我印象最深刻。

### 为什么印象深刻

**1. 技术挑战性高**

- 需要保证多个微服务间的数据一致性
- 要处理网络延迟、服务超时、部分失败等复杂情况
- 性能和一致性之间需要找到平衡点

**2. 解决方案的演进过程** 最初使用2PC（两阶段提交），但发现性能瓶颈严重，后来改用Saga模式：

```java
@Service
public class OrderSagaService {
    
    @Autowired
    private InventoryService inventoryService;
    
    @Autowired
    private PaymentService paymentService;
    
    public void processOrder(OrderRequest request) {
        SagaTransaction saga = SagaTransaction.builder()
            .step(this::reserveInventory, this::cancelInventoryReservation)
            .step(this::createOrder, this::cancelOrder)
            .step(this::processPayment, this::refundPayment)
            .step(this::addPoints, this::deductPoints)
            .build();
            
        saga.execute(request);
    }
}
```

**3. 学到的关键经验**

- 分布式系统中没有完美的解决方案，只有合适的权衡
- 业务补偿机制的设计比技术实现更重要
- 监控和可观测性在分布式系统中至关重要

### 延伸知识点

**分布式事务的几种模式：**

- **2PC/3PC**：强一致性，但性能差，存在单点故障
- **TCC模式**：Try-Confirm-Cancel，业务侵入性强但控制力强
- **Saga模式**：最终一致性，适合长事务链路
- **消息队列模式**：基于可靠消息的最终一致性

**实际项目中的选择标准：**

1. 业务对一致性的要求程度
2. 系统的性能要求
3. 团队的技术栈和维护能力
4. 错误处理和监控的复杂度

这个项目让我深刻理解了分布式系统的复杂性，也让我明白了架构设计不仅要考虑技术实现，更要考虑业务场景和团队能力。
