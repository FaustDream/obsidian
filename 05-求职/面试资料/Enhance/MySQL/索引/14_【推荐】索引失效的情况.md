# 索引失效的情况

## 数据库索引失效的常见情况

索引失效是数据库性能优化中的重要概念，了解这些情况有助于编写高效的SQL查询。

### 1. 查询条件导致的索引失效

**使用函数或表达式**

```sql
-- 索引失效：对索引字段使用函数
SELECT * FROM users WHERE UPPER(name) = 'JOHN';
SELECT * FROM orders WHERE YEAR(create_time) = 2024;

-- 正确写法：避免对索引字段使用函数
SELECT * FROM users WHERE name = 'JOHN';
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
```

**类型转换**

```sql
-- 索引失效：隐式类型转换
SELECT * FROM users WHERE phone = 13800138000;  -- phone是varchar类型

-- 正确写法：保持类型一致
SELECT * FROM users WHERE phone = '13800138000';
```

### 2. LIKE查询的索引失效

**前导通配符**

```sql
-- 索引失效：以%开头的LIKE查询
SELECT * FROM users WHERE name LIKE '%john%';
SELECT * FROM users WHERE name LIKE '%john';

-- 可以使用索引：以固定字符开头
SELECT * FROM users WHERE name LIKE 'john%';
```

### 3. 复合索引的最左前缀原则

```sql
-- 假设有复合索引：INDEX(a, b, c)

-- 可以使用索引的情况
SELECT * FROM table WHERE a = 1;
SELECT * FROM table WHERE a = 1 AND b = 2;
SELECT * FROM table WHERE a = 1 AND b = 2 AND c = 3;
SELECT * FROM table WHERE a = 1 AND c = 3;  -- 只有a部分使用索引

-- 无法使用索引的情况
SELECT * FROM table WHERE b = 2;
SELECT * FROM table WHERE c = 3;
SELECT * FROM table WHERE b = 2 AND c = 3;
```

### 4. 范围查询后的索引失效

```sql
-- 复合索引：INDEX(status, create_time, user_id)

-- 范围查询后的字段无法使用索引
SELECT * FROM orders 
WHERE status = 1 
  AND create_time > '2024-01-01'  -- 范围查询
  AND user_id = 100;  -- 这个条件无法使用索引
```

### 5. OR条件的索引失效

```sql
-- 索引失效：OR连接的条件，如果其中一个字段没有索引
SELECT * FROM users WHERE name = 'john' OR age = 25;  -- age没有索引

-- 解决方案：使用UNION
SELECT * FROM users WHERE name = 'john'
UNION
SELECT * FROM users WHERE age = 25;
```

### 6. NOT、!=、<>操作符

```sql
-- 这些操作符通常不会使用索引
SELECT * FROM users WHERE name != 'john';
SELECT * FROM users WHERE name <> 'john';
SELECT * FROM users WHERE NOT name = 'john';
```

### 7. IS NULL和IS NOT NULL

```sql
-- 在某些数据库中可能不使用索引
SELECT * FROM users WHERE name IS NULL;
SELECT * FROM users WHERE name IS NOT NULL;
```

### 8. 数据分布导致的索引失效

**选择性较差的情况**

```sql
-- 如果性别字段只有'男'和'女'两个值，索引效果很差
SELECT * FROM users WHERE gender = '男';  -- 可能不使用索引
```

**数据量过小**

```sql
-- 表中数据很少时，MySQL可能选择全表扫描而不是索引
SELECT * FROM small_table WHERE id = 1;  -- 可能不使用索引
```

### 9. 优化建议

**1. 查询优化**

```sql
-- 避免在WHERE子句中使用函数
-- 使用覆盖索引减少回表
-- 合理使用复合索引的顺序
```

**2. 索引设计原则**

```java
// 在Java中使用JPA时的索引注解
@Entity
@Table(name = "users")
@Index(name = "idx_name_age", columnList = "name, age")  // 复合索引
public class User {
    @Column(name = "name")
    private String name;
    
    @Column(name = "age")
    private Integer age;
}
```

**3. 使用EXPLAIN分析**

```sql
EXPLAIN SELECT * FROM users WHERE name = 'john';
```

### 10. 相关知识扩展

**MySQL索引类型**

- B+Tree索引（最常用）
- Hash索引
- 全文索引
- 空间索引

**索引优化策略**

- 选择合适的索引类型
- 定期分析索引使用情况
- 删除冗余索引
- 考虑索引的维护成本

理解这些索引失效的情况有助于编写更高效的SQL查询，在实际开发中应该结合具体的业务场景和数据特点来选择合适的索引策略。
