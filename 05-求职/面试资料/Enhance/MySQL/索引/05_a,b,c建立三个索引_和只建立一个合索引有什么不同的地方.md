# a,b,c建立三个索引，和只建立一个合索引有什么不同的地方

我来详细解释一下建立三个单独索引和一个复合索引的区别：

## 存储空间差异

**三个单独索引**：

- 每个索引都需要独立的存储空间
- 总体占用空间更大（约为复合索引的2-3倍）
- 每个索引都有自己的B+树结构

**一个复合索引**：

- 只需要一个B+树结构存储
- 空间占用相对较小
- 索引项按照字段顺序排列（如a,b,c）

## 查询性能差异

**使用场景分析**：

1. **查询条件为 WHERE a = ? AND b = ? AND c = ?**
    
    - 复合索引：直接命中，性能最优
    - 单独索引：需要进行索引合并操作，性能较差
2. **查询条件为 WHERE a = ?**
    
    - 复合索引：可以使用（符合最左前缀原则）
    - 单独索引：直接使用a索引，性能相当
3. **查询条件为 WHERE b = ?**
    
    - 复合索引：无法使用（不符合最左前缀原则）
    - 单独索引：可以使用b索引

## 索引维护成本

**单独索引**：

- INSERT/UPDATE/DELETE时需要维护三个索引
- 维护成本较高
- 并发写入时锁竞争可能更激烈

**复合索引**：

- 只需要维护一个索引结构
- 维护成本相对较低
- 写入性能更好

## 查询优化器选择

**MySQL优化器行为**：

```sql
-- 有三个单独索引时
EXPLAIN SELECT * FROM table WHERE a = 1 AND b = 2 AND c = 3;
-- 可能选择其中一个索引，然后回表过滤其他条件

-- 有复合索引时
EXPLAIN SELECT * FROM table WHERE a = 1 AND b = 2 AND c = 3;
-- 直接使用复合索引，无需回表
```

## 实际应用建议

**推荐使用复合索引的场景**：

- 经常同时查询多个字段
- 查询模式相对固定
- 写入频率不是特别高

**推荐使用单独索引的场景**：

- 查询模式多样化，各字段独立查询频繁
- 某些字段的选择性很好，单独使用效果佳
- 需要灵活的查询组合

## 延伸知识点

**索引合并（Index Merge）**：

- MySQL可以将多个单独索引的结果进行合并
- 但性能通常不如直接使用复合索引
- 包括intersection、union、sort-union等方式

**覆盖索引**：

- 复合索引更容易形成覆盖索引
- 避免回表操作，提升查询性能

**最左前缀原则**：

- 复合索引(a,b,c)可以支持a、(a,b)、(a,b,c)的查询
- 但不能支持单独的b或c查询

总的来说，复合索引在大多数场景下都是更好的选择，除非你的查询模式非常多样化。
