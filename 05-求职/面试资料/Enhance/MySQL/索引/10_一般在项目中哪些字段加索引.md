# 一般在项目中哪些字段加索引

在实际项目中，索引的添加需要根据具体的业务场景和查询模式来决定。以下是一些通用的索引添加原则和常见场景：

## 必须添加索引的字段

**主键字段**

- 所有表的主键（通常是id字段）会自动创建唯一索引
- 这是数据库的基本要求

**外键字段**

- 关联其他表的外键字段，如user_id、order_id等
- 提高JOIN查询性能

## 常见的索引场景

**频繁查询的字段**

```sql
-- 用户登录场景
CREATE INDEX idx_user_username ON user(username);
CREATE INDEX idx_user_email ON user(email);

-- 订单查询场景
CREATE INDEX idx_order_user_id ON order(user_id);
CREATE INDEX idx_order_status ON order(status);
```

**WHERE条件中的字段**

```sql
-- 根据状态查询
SELECT * FROM order WHERE status = 'PAID';

-- 根据创建时间范围查询
SELECT * FROM order WHERE created_time BETWEEN '2024-01-01' AND '2024-12-31';
```

**ORDER BY排序字段**

```sql
-- 按创建时间排序
CREATE INDEX idx_order_created_time ON order(created_time);
```

**GROUP BY分组字段**

```sql
-- 按用户分组统计
CREATE INDEX idx_order_user_id ON order(user_id);
```

## 业务场景举例

**电商系统**

- 商品表：category_id（分类查询）、price（价格筛选）、sales_count（销量排序）
- 订单表：user_id（用户订单）、status（订单状态）、created_time（时间范围）
- 用户表：phone（手机号登录）、email（邮箱登录）

**内容管理系统**

- 文章表：author_id（作者文章）、category_id（分类文章）、publish_time（发布时间）
- 评论表：article_id（文章评论）、user_id（用户评论）

## 组合索引的使用

**最左前缀原则**

```sql
-- 创建组合索引
CREATE INDEX idx_order_user_status_time ON order(user_id, status, created_time);

-- 可以使用索引的查询
SELECT * FROM order WHERE user_id = 1;
SELECT * FROM order WHERE user_id = 1 AND status = 'PAID';
SELECT * FROM order WHERE user_id = 1 AND status = 'PAID' AND created_time > '2024-01-01';
```

## 索引添加的注意事项

**避免过度索引**

- 每个索引都会占用存储空间
- 增加DML操作（INSERT、UPDATE、DELETE）的开销
- 一般建议单表索引数量控制在5-10个以内

**选择合适的索引类型**

- 普通索引：一般查询场景
- 唯一索引：需要保证数据唯一性
- 部分索引：只对部分数据建索引（PostgreSQL支持）

**监控索引效果**

```sql
-- MySQL查看索引使用情况
SHOW INDEX FROM table_name;
EXPLAIN SELECT * FROM table_name WHERE condition;
```

## 实际项目经验

**定期审查索引**

- 使用数据库监控工具分析慢查询
- 删除长期未使用的索引
- 根据新的业务需求调整索引策略

**测试环境验证**

- 在测试环境中模拟生产数据量
- 验证索引对查询性能的提升效果
- 确保索引不会对写入性能造成严重影响

总的来说，索引的添加应该基于实际的查询模式和业务需求，既要保证查询性能，又要控制索引的数量和维护成本。
