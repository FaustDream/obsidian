# 啥时候要建索引

## 什么时候需要建立索引

索引是数据库优化的重要手段，但并不是越多越好。以下是建立索引的主要场景：

### 需要建立索引的情况

**1. 频繁查询的字段**

- WHERE子句中经常使用的字段
- JOIN操作中的连接字段
- ORDER BY和GROUP BY中的字段

```sql
-- 经常这样查询，就需要在user_id上建索引
SELECT * FROM orders WHERE user_id = 123;

-- 经常按创建时间排序，就需要在created_at上建索引
SELECT * FROM orders ORDER BY created_at DESC;
```

**2. 唯一性约束字段**

- 主键（自动创建）
- 唯一键字段
- 业务上需要保证唯一性的字段

**3. 外键字段**

- 提高关联查询性能
- 加速JOIN操作

### 不建议建立索引的情况

**1. 频繁更新的字段**

- 索引会降低INSERT/UPDATE/DELETE性能
- 每次数据变更都需要维护索引

**2. 数据量很小的表**

- 全表扫描可能比索引查找更快
- 通常指几千条以下的数据

**3. 选择性很低的字段**

- 性别字段（只有男/女两个值）
- 状态字段（只有几个固定值）
- 这类字段建索引效果不明显

```sql
-- 不建议对性别字段建索引
SELECT * FROM users WHERE gender = 'M'; -- 可能返回50%的数据
```

### 索引使用的经验法则

**1. 复合索引的顺序很重要**

```sql
-- 如果经常这样查询
SELECT * FROM users WHERE city = 'Beijing' AND age = 25;

-- 建议创建复合索引
CREATE INDEX idx_city_age ON users(city, age);
-- 而不是
CREATE INDEX idx_age_city ON users(age, city);
```

**2. 最左前缀原则**

- 复合索引(a,b,c)可以支持(a)、(a,b)、(a,b,c)的查询
- 但不能有效支持(b)、(c)、(b,c)的查询

**3. 覆盖索引**

```sql
-- 如果经常需要查询user_id和created_at
SELECT user_id, created_at FROM orders WHERE status = 'completed';

-- 可以创建覆盖索引
CREATE INDEX idx_status_userid_created ON orders(status, user_id, created_at);
```

### 实际开发中的建议

**1. 监控慢查询**

- 通过慢查询日志发现性能问题
- 针对性地为慢查询涉及的字段建索引

**2. 分析查询计划**

```sql
-- MySQL中使用EXPLAIN分析查询
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';
```

**3. 定期维护索引**

- 删除无用的索引
- 重建碎片化严重的索引
- 监控索引使用情况

记住：索引是一把双刃剑，既能提高查询性能，也会占用存储空间并影响写入性能。关键是要根据实际的业务查询模式来合理设计索引策略。
