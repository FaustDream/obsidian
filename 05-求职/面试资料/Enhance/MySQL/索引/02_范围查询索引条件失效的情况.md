MySQL中范围查询后的索引条件失效是一个重要的查询优化问题。这主要涉及**联合索引的最左前缀原则**和**索引条件下推**的机制。

## 核心原理

在联合索引中，当遇到范围查询（如 `>`、`<`、`BETWEEN`、`LIKE 'abc%'`）时，MySQL无法继续使用该范围查询字段后面的索引字段进行精确匹配，这些后续字段的索引会"失效"。

## 典型场景示例

假设有一个联合索引：`INDEX(a, b, c)`

```sql
-- 情况1：b字段范围查询后，c字段索引失效
SELECT * FROM table WHERE a = 1 AND b > 10 AND c = 'value';
-- 实际只能使用索引 (a, b)，c字段需要回表后过滤

-- 情况2：完全可以使用索引
SELECT * FROM table WHERE a = 1 AND b = 10 AND c = 'value';
-- 可以使用完整索引 (a, b, c)
```

## 具体失效情况

### 1. 范围查询类型

- `>`, `<`, `>=`, `<=`
- `BETWEEN ... AND ...`
- `LIKE 'abc%'`（前缀匹配）
- `IN` 在某些情况下也会被视为范围查询

### 2. 索引使用分析

```sql
-- 假设索引：INDEX(status, create_time, user_id)

-- 案例1：create_time范围查询导致user_id索引失效
SELECT * FROM orders 
WHERE status = 1 
  AND create_time > '2023-01-01' 
  AND user_id = 12345;
-- 只能使用 (status, create_time)

-- 案例2：调整查询顺序也无法改变
SELECT * FROM orders 
WHERE status = 1 
  AND user_id = 12345 
  AND create_time > '2023-01-01';
-- 仍然只能使用 (status, create_time)
```

## 优化策略

### 1. 调整索引字段顺序

将范围查询字段放在索引的最后：

```sql
-- 原索引：INDEX(status, create_time, user_id)
-- 优化为：INDEX(status, user_id, create_time)

SELECT * FROM orders 
WHERE status = 1 
  AND user_id = 12345 
  AND create_time > '2023-01-01';
-- 现在可以使用完整索引
```

### 2. 创建多个索引

根据不同查询模式创建不同的索引：

```sql
-- 针对不同查询创建专门索引
INDEX(status, create_time)  -- 用于时间范围查询
INDEX(status, user_id)      -- 用于用户精确查询
```

### 3. 使用覆盖索引

将查询需要的字段都包含在索引中，避免回表：

```sql
-- 如果只需要这几个字段
INDEX(status, create_time, user_id, order_amount)
```

## 注意事项

1. **MySQL 5.6+的索引条件下推**：即使索引"失效"，MySQL仍会在存储引擎层面使用这些条件进行过滤，减少回表次数。
    
2. **EXPLAIN分析**：使用 `EXPLAIN` 查看实际的索引使用情况，关注 `key_len` 字段来判断索引使用长度。
    
3. **业务场景权衡**：不是所有情况都需要优化，要根据查询频率和数据量来判断是否值得调整索引结构。
    

这个机制的理解对于MySQL查询优化非常重要，合理的索引设计可以显著提升查询性能。
