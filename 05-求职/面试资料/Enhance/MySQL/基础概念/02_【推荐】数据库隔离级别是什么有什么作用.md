# 数据库隔离级别是什么？有什么作用？

数据库隔离级别是数据库事务处理中的一个重要概念，它定义了一个事务与其他并发事务之间的隔离程度。

## 什么是数据库隔离级别

数据库隔离级别决定了当多个事务同时访问数据库时，事务之间如何相互影响。它是ACID特性中"隔离性(Isolation)"的具体体现。

## 四种标准隔离级别

### 1. READ UNCOMMITTED（读未提交）

- **特点**：最低的隔离级别，事务可以读取其他事务未提交的数据
- **问题**：会产生脏读、不可重复读、幻读
- **性能**：最高
- **使用场景**：几乎不使用，因为数据一致性无法保证

### 2. READ COMMITTED（读已提交）

- **特点**：只能读取已经提交的数据
- **解决问题**：避免脏读
- **仍存在问题**：不可重复读、幻读
- **性能**：较高
- **使用场景**：Oracle、SQL Server的默认级别

### 3. REPEATABLE READ（可重复读）

- **特点**：在同一事务中多次读取同一数据结果一致
- **解决问题**：避免脏读、不可重复读
- **仍存在问题**：幻读（理论上，MySQL的InnoDB通过Next-Key Lock解决了幻读）
- **性能**：中等
- **使用场景**：MySQL InnoDB的默认级别

### 4. SERIALIZABLE（串行化）

- **特点**：最高隔离级别，事务串行执行
- **解决问题**：避免所有并发问题
- **性能**：最低
- **使用场景**：对数据一致性要求极高的场景

## 并发问题详解

### 脏读（Dirty Read）

```java
// 事务A
UPDATE account SET balance = 1000 WHERE id = 1;
// 未提交

// 事务B
SELECT balance FROM account WHERE id = 1; // 读到1000
// 如果事务A回滚，事务B读到的就是脏数据
```

### 不可重复读（Non-Repeatable Read）

> 我觉得称为 "没法重复读" 更合适，相当于只能读取一次。

```java
// 事务A
SELECT balance FROM account WHERE id = 1; // 读到500

// 事务B
UPDATE account SET balance = 1000 WHERE id = 1;
COMMIT;

// 事务A
SELECT balance FROM account WHERE id = 1; // 读到1000，前后不一致
```

### 幻读（Phantom Read）

```java
// 事务A
SELECT COUNT(*) FROM account WHERE balance > 100; // 得到5条

// 事务B
INSERT INTO account VALUES(6, 200);
COMMIT;

// 事务A
SELECT COUNT(*) FROM account WHERE balance > 100; // 得到6条，出现"幻影"
```

## 实际应用中的考虑

### 1. 性能与一致性的权衡

- 隔离级别越高，并发性能越低
- 需要根据业务需求选择合适的级别

### 2. 不同数据库的实现差异

```java
// MySQL设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

// 查看当前隔离级别
SELECT @@tx_isolation;
```

### 3. 在Java中的应用

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    // 转账逻辑
}
```

## 扩展知识

### MVCC（多版本并发控制）

现代数据库如MySQL InnoDB通过MVCC机制在保证隔离性的同时提升并发性能，每个事务看到的是数据的一个快照版本。

### 锁机制

- 共享锁（S锁）：读锁
- 排他锁（X锁）：写锁
- 意向锁：表级锁，提高锁检测效率

### 死锁处理

数据库会检测死锁并回滚其中一个事务，应用程序需要处理这种情况。

理解隔离级别对于设计高并发系统至关重要，它直接影响系统的性能和数据一致性。在实际项目中，需要根据具体业务场景选择合适的隔离级别。
