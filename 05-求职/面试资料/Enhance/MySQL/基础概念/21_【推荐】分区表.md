## MySQL 分区表详解

### 什么是分区表？

MySQL分区表是将一个大表在**物理存储层面**分割成多个较小的、更易管理的部分，但在**逻辑层面**仍然是一个完整的表。这就像把一本厚书按章节分成几本小册子，但读者仍然认为这是同一本书。

### 分区的核心原理

```sql
-- 创建分区表示例
CREATE TABLE orders (
    id INT NOT NULL,
    order_date DATE NOT NULL,
    customer_id INT,
    amount DECIMAL(10,2)
) PARTITION BY RANGE (YEAR(order_date)) (
    PARTITION p2020 VALUES LESS THAN (2021),
    PARTITION p2021 VALUES LESS THAN (2022),
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024)
);
```

### 分区类型详解

#### 1. RANGE分区（范围分区）

```sql
-- 按日期范围分区
PARTITION BY RANGE (YEAR(created_date)) (
    PARTITION p_old VALUES LESS THAN (2020),
    PARTITION p_2020 VALUES LESS THAN (2021),
    PARTITION p_2021 VALUES LESS THAN (2022)
);
```

#### 2. LIST分区（列表分区）

```sql
-- 按具体值分区
PARTITION BY LIST (region) (
    PARTITION p_north VALUES IN ('北京','天津','河北'),
    PARTITION p_south VALUES IN ('广东','深圳','香港'),
    PARTITION p_west VALUES IN ('四川','重庆','云南')
);
```

#### 3. HASH分区（哈希分区）

```sql
-- 按哈希值均匀分布
PARTITION BY HASH(customer_id) PARTITIONS 4;
```

#### 4. KEY分区

```sql
-- 使用MySQL内置哈希函数
PARTITION BY KEY(id) PARTITIONS 4;
```

### 分区表的优势

**1. 查询性能提升**

- **分区消除**：只扫描相关分区，大幅减少I/O
- **并行处理**：多个分区可同时执行查询

**2. 数据管理便利**

```sql
-- 快速删除历史数据
ALTER TABLE orders DROP PARTITION p2020;

-- 添加新分区
ALTER TABLE orders ADD PARTITION (
    PARTITION p2024 VALUES LESS THAN (2025)
);
```

**3. 维护效率**

- 分区级别的备份和恢复
- 独立的索引维护
- 更快的数据导入导出

### 实际应用场景

**时间序列数据**

```sql
-- 日志表按月分区
CREATE TABLE access_logs (
    log_time DATETIME,
    user_id INT,
    action VARCHAR(100)
) PARTITION BY RANGE (MONTH(log_time)) (
    PARTITION p_jan VALUES LESS THAN (2),
    PARTITION p_feb VALUES LESS THAN (3),
    -- ...
);
```

**地理数据分布**

```sql
-- 订单表按地区分区
CREATE TABLE regional_orders (
    order_id INT,
    region_code INT,
    amount DECIMAL(10,2)
) PARTITION BY LIST (region_code) (
    PARTITION p_华北 VALUES IN (10,11,12),
    PARTITION p_华南 VALUES IN (20,21,22)
);
```

### 分区表的限制与注意事项

**技术限制**

1. 分区键必须包含在主键中
2. 外键支持有限
3. 全文索引不支持
4. 最大分区数：8192个

**设计原则**

```sql
-- ❌ 错误示例：分区键不在主键中
CREATE TABLE bad_example (
    id INT PRIMARY KEY,
    create_date DATE
) PARTITION BY RANGE (YEAR(create_date));

-- ✅ 正确示例：分区键包含在复合主键中
CREATE TABLE good_example (
    id INT,
    create_date DATE,
    PRIMARY KEY (id, create_date)
) PARTITION BY RANGE (YEAR(create_date));
```

### 性能优化最佳实践

**1. 分区消除优化**

```sql
-- 确保WHERE条件能触发分区消除
SELECT * FROM orders 
WHERE order_date >= '2023-01-01' 
  AND order_date < '2024-01-01';
```

**2. 合理的分区数量**

- 避免过多分区（建议<100个）
- 每个分区数据量适中（几百万到几千万行）

**3. 监控分区效果**

```sql
-- 查看分区信息
SELECT 
    PARTITION_NAME,
    TABLE_ROWS,
    DATA_LENGTH/1024/1024 as DATA_MB
FROM information_schema.PARTITIONS 
WHERE TABLE_NAME = 'orders';
```

### 与分库分表的区别

|特性|分区表|分库分表|
|---|---|---|
|复杂度|低（MySQL自动管理）|高（应用层处理）|
|跨分区查询|透明支持|需要中间件|
|扩展性|单机限制|可横向扩展|
|事务支持|完全支持|分布式事务复杂|

### 总结

MySQL分区表是处理大数据量的有效技术方案，特别适合有明确分区逻辑的场景（如时间、地域）。它在保持SQL简单性的同时，显著提升了查询性能和数据管理效率。但需要注意分区键的选择和相关技术限制，在实际项目中要根据具体业务场景来权衡是否采用分区策略。
