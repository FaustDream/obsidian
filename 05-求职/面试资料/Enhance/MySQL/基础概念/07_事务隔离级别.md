# 事务隔离级别

事务隔离级别是数据库系统中控制并发事务之间相互影响程度的重要概念。让我为您详细解释：

## 四种标准隔离级别

### 1. 读未提交 (Read Uncommitted)

- **最低隔离级别**
- 事务可以读取其他未提交事务的数据
- **问题**：会出现脏读、不可重复读、幻读
- **性能**：最高，但数据一致性最差

### 2. 读已提交 (Read Committed)

- 事务只能读取已经提交的数据
- **解决**：脏读问题
- **仍存在**：不可重复读、幻读
- **应用**：Oracle、SQL Server默认级别

### 3. 可重复读 (Repeatable Read)

- 在同一事务中多次读取同一数据结果相同
- **解决**：脏读、不可重复读
- **仍存在**：幻读（部分数据库如MySQL InnoDB通过间隙锁解决）
- **应用**：MySQL默认级别

### 4. 串行化 (Serializable)

- **最高隔离级别**
- 事务完全串行执行
- **解决**：所有并发问题
- **性能**：最低，但数据一致性最强

## 三种并发问题

```java
// 脏读示例
Transaction A: UPDATE account SET balance = 1000 WHERE id = 1;
Transaction B: SELECT balance FROM account WHERE id = 1; // 读到1000
Transaction A: ROLLBACK; // A回滚，B读到了脏数据

// 不可重复读示例
Transaction A: SELECT balance FROM account WHERE id = 1; // 读到500
Transaction B: UPDATE account SET balance = 1000 WHERE id = 1; COMMIT;
Transaction A: SELECT balance FROM account WHERE id = 1; // 读到1000，前后不一致

// 幻读示例
Transaction A: SELECT COUNT(*) FROM account WHERE balance > 100; // 返回5条
Transaction B: INSERT INTO account VALUES(6, 200); COMMIT;
Transaction A: SELECT COUNT(*) FROM account WHERE balance > 100; // 返回6条，出现幻行
```

## MySQL中的实现

### InnoDB存储引擎

```sql
-- 设置事务隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 查看当前隔离级别
SELECT @@tx_isolation;

-- MySQL 8.0后使用
SELECT @@transaction_isolation;
```

### MVCC (多版本并发控制)

InnoDB通过MVCC实现非锁定读：

- 每行数据有隐藏的版本号
- 读取时根据事务开始时间决定读取哪个版本
- 实现了在RR级别下避免幻读

## Java中的应用

### Spring事务管理

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    // 业务逻辑
}

// 隔离级别枚举
public enum Isolation {
    DEFAULT(-1),
    READ_UNCOMMITTED(1),
    READ_COMMITTED(2),
    REPEATABLE_READ(4),
    SERIALIZABLE(8);
}
```

### JDBC设置

```java
Connection conn = DriverManager.getConnection(url, user, password);
conn.setTransactionIsolation(Connection.TRANSACTION_READ_COMMITTED);
```

## 性能与一致性权衡

|隔离级别|脏读|不可重复读|幻读|性能|
|---|---|---|---|---|
|读未提交|✗|✗|✗|最高|
|读已提交|✓|✗|✗|较高|
|可重复读|✓|✓|✗|较低|
|串行化|✓|✓|✓|最低|

## 实际应用建议

1. **金融系统**：通常选择REPEATABLE_READ或SERIALIZABLE
2. **电商系统**：READ_COMMITTED较为常见
3. **报表系统**：可考虑READ_UNCOMMITTED（对一致性要求不高）
4. **高并发场景**：结合业务需求选择合适级别，避免过度隔离

## 扩展知识

### 死锁问题

```java
// 可能导致死锁的场景
Transaction A: 锁定资源1 → 等待资源2
Transaction B: 锁定资源2 → 等待资源1
```

### 锁的类型

- **共享锁(S锁)**：读锁，可并发
- **排它锁(X锁)**：写锁，互斥
- **意向锁**：表级锁，提高锁检查效率

理解事务隔离级别对于数据库性能优化和数据一致性保证都至关重要，需要根据具体业务场景选择合适的隔离级别。
