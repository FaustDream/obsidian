## 子查询优化示例详解

### 1. 性能瓶颈类优化

#### ❌ 问题：子查询返回大量数据

```sql
-- 低效：子查询可能返回大量用户数据
SELECT p.product_name, p.price
FROM products p
WHERE p.category_id IN (
    SELECT category_id 
    FROM categories 
    WHERE region = 'Asia'  -- 可能返回很多分类ID
);
```

#### ✅ 优化：使用JOIN

```sql
-- 高效：直接JOIN避免大结果集传递
SELECT p.product_name, p.price
FROM products p
JOIN categories c ON p.category_id = c.category_id
WHERE c.region = 'Asia';
```

### 2. 重复执行（相关子查询）优化

#### ❌ 问题：相关子查询重复执行

```sql
-- 低效：对每个用户都执行一次子查询
SELECT u.name, u.email,
       (SELECT COUNT(*) 
        FROM orders o 
        WHERE o.user_id = u.id) AS order_count
FROM users u
WHERE u.status = 'active';
```

#### ✅ 优化：使用LEFT JOIN + GROUP BY

```sql
-- 高效：一次JOIN完成
SELECT u.name, u.email, COALESCE(COUNT(o.id), 0) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.status = 'active'
GROUP BY u.id, u.name, u.email;
```

### 3. EXISTS/NOT EXISTS优化

#### ❌ 问题：NOT EXISTS子查询效率低

```sql
-- 低效：对每个产品检查是否有订单
SELECT p.product_name
FROM products p
WHERE NOT EXISTS (
    SELECT 1 
    FROM order_items oi 
    WHERE oi.product_id = p.id
);
```

#### ✅ 优化：使用LEFT JOIN + IS NULL

```sql
-- 高效：一次LEFT JOIN找出无订单产品
SELECT p.product_name
FROM products p
LEFT JOIN order_items oi ON p.id = oi.product_id
WHERE oi.product_id IS NULL;
```

### 4. 复杂嵌套子查询优化

#### ❌ 问题：多层嵌套复杂

```sql
-- 复杂：多层嵌套难以优化
SELECT u.name
FROM users u
WHERE u.id IN (
    SELECT o.user_id
    FROM orders o
    WHERE o.total > (
        SELECT AVG(total)
        FROM orders
        WHERE user_id IN (
            SELECT id 
            FROM users 
            WHERE registration_date > '2023-01-01'
        )
    )
);
```

#### ✅ 优化：使用CTE分步处理

```sql
-- 清晰：分步骤，易于优化
WITH new_users AS (
    SELECT id 
    FROM users 
    WHERE registration_date > '2023-01-01'
),
avg_order_total AS (
    SELECT AVG(total) as avg_total
    FROM orders o
    JOIN new_users nu ON o.user_id = nu.id
),
high_value_users AS (
    SELECT DISTINCT o.user_id
    FROM orders o
    CROSS JOIN avg_order_total aot
    WHERE o.total > aot.avg_total
)
SELECT u.name
FROM users u
JOIN high_value_users hvu ON u.id = hvu.user_id;
```

### 5. 聚合子查询优化

#### ❌ 问题：聚合子查询重复计算

```sql
-- 低效：每行都计算一次聚合
SELECT 
    product_name,
    price,
    price / (SELECT AVG(price) FROM products) as price_ratio
FROM products
WHERE category_id = 1;
```

#### ✅ 优化：预计算聚合值

```sql
-- 高效：聚合值只计算一次
SELECT 
    product_name,
    price,
    price / avg_price as price_ratio
FROM products
CROSS JOIN (SELECT AVG(price) as avg_price FROM products) avg_data
WHERE category_id = 1;
```

### 6. IN子查询优化

#### ❌ 问题：IN子查询可能效率低

```sql
-- 可能低效：特别是子查询结果集很大时
SELECT customer_name
FROM customers
WHERE customer_id IN (
    SELECT user_id
    FROM orders
    WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31'
);
```

#### ✅ 优化：使用EXISTS或JOIN

```sql
-- 方案1: EXISTS（通常更快）
SELECT customer_name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.user_id = c.customer_id
    AND o.order_date BETWEEN '2024-01-01' AND '2024-12-31'
);

-- 方案2: SEMI JOIN（MySQL 8.0+）
SELECT DISTINCT c.customer_name
FROM customers c
JOIN orders o ON c.customer_id = o.user_id
WHERE o.order_date BETWEEN '2024-01-01' AND '2024-12-31';
```

### 7. 窗口函数替代子查询

#### ❌ 问题：排名子查询复杂

```sql
-- 复杂：获取每个部门薪水最高的员工
SELECT e1.name, e1.department, e1.salary
FROM employees e1
WHERE e1.salary = (
    SELECT MAX(e2.salary)
    FROM employees e2
    WHERE e2.department = e1.department
);
```

#### ✅ 优化：使用窗口函数

```sql
-- 简洁：窗口函数一次完成
SELECT name, department, salary
FROM (
    SELECT name, department, salary,
           RANK() OVER (PARTITION BY department ORDER BY salary DESC) as rank_num
    FROM employees
) ranked
WHERE rank_num = 1;
```

## 优化效果对比

|场景|优化前|优化后|性能提升|
|---|---|---|---|
|相关子查询|O(n×m)|O(n+m)|显著|
|NOT EXISTS|多次扫描|一次LEFT JOIN|中等|
|IN大结果集|内存占用高|EXISTS扫描|显著|
|嵌套聚合|重复计算|预计算|中等|

记住：**先用EXPLAIN分析执行计划，再根据实际性能瓶颈选择合适的优化方案**。
