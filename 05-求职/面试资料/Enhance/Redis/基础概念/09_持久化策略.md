Redis提供了两种持久化策略，RDB（Redis Database Backup File）数据快照，AOF（Append Only File）日志追加。

RDB：把Redis内存中的数据周期性的写入磁盘生成快照文件；
> RDB的数据在两次备份之间会丢失，有压缩，快照文件体积小，宕机恢复速度快，系统资源占用高，适用于对数据完整性要求不高，但需要快速恢复的场景，定期备份数据，用于灾难恢复或数据迁移；

AOF：将redis执行的写命令以协议格式追加到AOF文件中；
> AOF数据安全性高，取决于刷盘策略，记录命令，文件体积大，宕机恢复速度慢，系统资源占用低，AOF重写时占用高，适用于对数据完整性高或需要实时持久化的场景；

Redis有两种主要的持久化策略，我来详细介绍一下：

## 1. RDB (Redis Database Backup)

**工作原理：**

- 在指定的时间间隔内，将内存中的数据快照写入磁盘
- 生成一个压缩的二进制文件（dump.rdb）
- 类似于数据库的"全量备份"

**触发方式：**

- **自动触发：** 通过配置文件设置条件，如`save 900 1`（900秒内至少1个key发生变化）
- **手动触发：** 执行`SAVE`或`BGSAVE`命令
- **其他触发：** 主从复制、shutdown时也会触发

**优点：**

- 文件紧凑，适合备份和灾难恢复
- 恢复速度快，直接加载整个数据集
- 对Redis性能影响相对较小（使用子进程）

**缺点：**

- 数据安全性较差，可能丢失最后一次快照后的数据
- 数据量大时，fork子进程会比较耗时

## 2. AOF (Append Only File)

**工作原理：**

- 记录每一个写操作命令到日志文件
- Redis重启时重新执行AOF文件中的命令来恢复数据
- 类似于数据库的"增量备份"

**同步策略：**

```
# 配置选项
appendfsync always    # 每个写命令都同步，最安全但性能最差
appendfsync everysec  # 每秒同步一次，默认配置，平衡性能和安全
appendfsync no        # 由操作系统决定何时同步，性能最好但最不安全
```

**AOF重写机制：**

- 当AOF文件过大时，Redis会创建新的AOF文件
- 只保留恢复当前数据集所需的最小命令集合
- 可通过`BGREWRITEAOF`手动触发

**优点：**

- 数据安全性更高，最多丢失1秒数据（everysec模式）
- 文件内容可读，便于分析和修复
- 支持秒级数据恢复

**缺点：**

- 文件体积通常比RDB大
- 恢复速度相对较慢
- 对性能影响较大

## 3. 混合持久化（Redis 4.0+）

**特点：**

- 结合RDB和AOF的优点
- AOF重写时，先写入RDB格式的数据，再追加增量的AOF数据
- 兼顾了恢复速度和数据安全性

**配置：**

```
aof-use-rdb-preamble yes  # 开启混合持久化
```

## 实际使用建议

**生产环境推荐配置：**

```redis
# 同时开启两种持久化
save 900 1
save 300 10
save 60 10000

appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
```

**选择策略：**

- **高可用性要求：** 选择AOF + 混合持久化
- **性能优先：** 选择RDB
- **平衡考虑：** 同时使用RDB和AOF

**相关面试延伸点：**

1. Redis的fork机制和Copy-on-Write
2. 主从复制中的持久化策略
3. Redis Cluster环境下的持久化考虑
4. 持久化对内存使用的影响
5. 不同业务场景下的持久化选择

这样的持久化策略设计体现了Redis在性能和数据安全性之间的权衡思考，也是很多分布式系统需要面对的经典问题。
