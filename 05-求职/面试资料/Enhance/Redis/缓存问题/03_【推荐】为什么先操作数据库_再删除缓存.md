# 为什么先操作数据库，再删除缓存

这是一个关于缓存一致性策略的经典问题。让我详细解释为什么推荐"先操作数据库，再删除缓存"这种模式。

## 核心原因分析

### 1. 数据一致性保障

先操作数据库的主要原因是**确保数据的最终一致性**。如果先删除缓存再操作数据库，可能出现以下问题：

```java
// 错误的做法：先删缓存，再操作数据库
public void updateUser(User user) {
    // 1. 先删除缓存
    redisTemplate.delete("user:" + user.getId());
    
    // 2. 如果这里数据库操作失败
    userMapper.updateUser(user); // 可能抛出异常
    
    // 结果：缓存被删除，但数据库没有更新，数据不一致
}
```

### 2. 避免脏数据读取

考虑这个并发场景：

```java
// 线程A：更新操作
Thread A: 删除缓存 -> 数据库更新中...

// 线程B：查询操作  
Thread B: 查询缓存(miss) -> 查询数据库(旧数据) -> 写入缓存(脏数据)

// 线程A：数据库更新完成
Thread A: 数据库更新完成

// 结果：缓存中是旧数据，数据库是新数据
```

## 推荐的实现方案

### 1. 基础版本：先更新数据库，再删除缓存

```java
@Service
public class UserService {
    
    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @Transactional
    public void updateUser(User user) {
        // 1. 先更新数据库
        userMapper.updateUser(user);
        
        // 2. 再删除缓存
        String cacheKey = "user:" + user.getId();
        redisTemplate.delete(cacheKey);
    }
}
```

### 2. 增强版本：加入异常处理和重试机制

```java
@Service
public class UserService {
    
    @Retryable(value = {Exception.class}, maxAttempts = 3)
    public void updateUser(User user) {
        // 1. 更新数据库
        userMapper.updateUser(user);
        
        // 2. 删除缓存（即使失败也不影响数据库事务）
        try {
            deleteCacheAsync(user.getId());
        } catch (Exception e) {
            // 记录日志，可以后续补偿
            log.error("删除缓存失败，用户ID: {}", user.getId(), e);
        }
    }
    
    @Async
    private void deleteCacheAsync(Long userId) {
        String cacheKey = "user:" + userId;
        redisTemplate.delete(cacheKey);
    }
}
```

## 延伸知识点

### 1. 缓存双删策略

为了进一步提高一致性，可以采用双删策略：

```java
public void updateUserWithDoubleDelete(User user) {
    String cacheKey = "user:" + user.getId();
    
    // 第一次删除
    redisTemplate.delete(cacheKey);
    
    // 更新数据库
    userMapper.updateUser(user);
    
    // 延迟第二次删除
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(1000); // 延迟1秒
            redisTemplate.delete(cacheKey);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}
```

### 2. 分布式锁保证原子性

```java
@Service
public class UserService {
    
    public void updateUserWithLock(User user) {
        String lockKey = "lock:user:" + user.getId();
        RLock lock = redissonClient.getLock(lockKey);
        
        try {
            if (lock.tryLock(10, TimeUnit.SECONDS)) {
                // 更新数据库
                userMapper.updateUser(user);
                
                // 删除缓存
                String cacheKey = "user:" + user.getId();
                redisTemplate.delete(cacheKey);
            }
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 3. 基于MQ的最终一致性

```java
@Service
public class UserService {
    
    @Autowired
    private RabbitTemplate rabbitTemplate;
    
    @Transactional
    public void updateUser(User user) {
        // 1. 更新数据库
        userMapper.updateUser(user);
        
        // 2. 发送删除缓存消息
        CacheDeleteMessage message = new CacheDeleteMessage();
        message.setCacheKey("user:" + user.getId());
        message.setOperation("DELETE");
        
        rabbitTemplate.convertAndSend("cache.delete.queue", message);
    }
}

@RabbitListener(queues = "cache.delete.queue")
@Component
public class CacheDeleteListener {
    
    @RabbitHandler
    public void handleCacheDelete(CacheDeleteMessage message) {
        try {
            redisTemplate.delete(message.getCacheKey());
        } catch (Exception e) {
            // 重试或记录失败日志
            log.error("删除缓存失败: {}", message.getCacheKey(), e);
        }
    }
}
```

## 总结

"先操作数据库，再删除缓存"是目前最佳实践，因为：

1. **数据库是主要数据源**，优先保证数据库操作成功
2. **缓存删除失败的影响相对较小**，最多是短暂的数据不一致
3. **避免了缓存删除成功但数据库操作失败**的严重问题
4. **可以通过异步删除、重试机制等方式进一步优化**

这种模式在高并发场景下能够很好地平衡性能和一致性需求。
