# 缓存不一致的问题，删除失败处理

缓存不一致问题是分布式系统中的经典难题，特别是在删除缓存失败时需要特别谨慎处理。让我详细解析这个问题：

## 缓存不一致的根本原因

缓存不一致主要发生在数据库和缓存的双写场景中，常见的更新策略有：

- **先更新数据库，再删除缓存**（推荐）
- **先删除缓存，再更新数据库**
- **先更新缓存，再更新数据库**（不推荐）

## 删除失败的处理策略

### 1. 重试机制

```java
@Service
public class CacheService {
    
    @Retryable(value = {Exception.class}, maxAttempts = 3, backoff = @Backoff(delay = 1000))
    public void deleteCache(String key) {
        try {
            redisTemplate.delete(key);
        } catch (Exception e) {
            log.error("缓存删除失败，key: {}", key, e);
            throw e; // 重新抛出异常触发重试
        }
    }
    
    // 重试失败后的兜底处理
    @Recover
    public void recover(Exception ex, String key) {
        // 将失败的删除操作放入消息队列异步处理
        messageQueue.send(new CacheDeleteMessage(key));
    }
}
```

### 2. 异步补偿机制

```java
@Component
public class CacheConsistencyHandler {
    
    public void updateDataWithCache(Long id, UserData userData) {
        // 1. 先更新数据库
        userRepository.updateById(id, userData);
        
        // 2. 异步删除缓存
        CompletableFuture.runAsync(() -> {
            try {
                cacheService.deleteCache("user:" + id);
            } catch (Exception e) {
                // 删除失败，发送到延迟队列
                delayedMessageQueue.send(
                    new CacheDeleteMessage("user:" + id), 
                    Duration.ofSeconds(30)
                );
            }
        });
    }
}
```

### 3. 基于Canal的数据同步

```java
@Component
public class CanalCacheHandler {
    
    @EventListener
    public void handleDatabaseChange(CanalEntry.Entry entry) {
        if (entry.getEntryType() == CanalEntry.EntryType.ROWDATA) {
            CanalEntry.RowChange rowChange = CanalEntry.RowChange.parseFrom(entry.getStoreValue());
            
            for (CanalEntry.RowData rowData : rowChange.getRowDatasList()) {
                String tableName = entry.getHeader().getTableName();
                String primaryKey = extractPrimaryKey(rowData);
                
                // 根据数据库变更删除对应缓存
                String cacheKey = tableName + ":" + primaryKey;
                cacheService.deleteCache(cacheKey);
            }
        }
    }
}
```

### 4. 设置缓存过期时间作为兜底

```java
@Service
public class UserService {
    
    public UserData getUserById(Long id) {
        String cacheKey = "user:" + id;
        
        // 从缓存获取
        UserData cached = cacheService.get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // 缓存未命中，查询数据库
        UserData userData = userRepository.findById(id);
        
        // 设置缓存，添加过期时间作为最后的数据一致性保障
        cacheService.setWithExpire(cacheKey, userData, Duration.ofMinutes(30));
        
        return userData;
    }
}
```

## 最佳实践方案

### 完整的缓存更新流程

```java
@Service
@Transactional
public class UserServiceImpl implements UserService {
    
    public void updateUser(Long id, UserData userData) {
        try {
            // 1. 数据库操作
            userRepository.updateById(id, userData);
            
            // 2. 注册事务后置处理器，确保事务提交后再删除缓存
            TransactionSynchronizationManager.registerSynchronization(
                new TransactionSynchronization() {
                    @Override
                    public void afterCommit() {
                        handleCacheDelete("user:" + id);
                    }
                }
            );
            
        } catch (Exception e) {
            log.error("用户更新失败", e);
            throw e;
        }
    }
    
    private void handleCacheDelete(String cacheKey) {
        try {
            // 立即删除缓存
            cacheService.deleteCache(cacheKey);
        } catch (Exception e) {
            // 删除失败，多重保障
            log.warn("缓存删除失败，启用补偿机制，key: {}", cacheKey);
            
            // 1. 重试队列
            retryQueue.add(new CacheDeleteTask(cacheKey, System.currentTimeMillis()));
            
            // 2. 设置一个短期的脏数据标记
            cacheService.set(cacheKey + ":dirty", "1", Duration.ofMinutes(5));
        }
    }
}
```

### 读取时的脏数据检测

```java
public UserData getUserById(Long id) {
    String cacheKey = "user:" + id;
    String dirtyKey = cacheKey + ":dirty";
    
    // 检查是否有脏数据标记
    if (cacheService.exists(dirtyKey)) {
        // 有脏数据标记，强制从数据库读取
        UserData userData = userRepository.findById(id);
        cacheService.delete(cacheKey); // 清除可能的脏缓存
        cacheService.delete(dirtyKey); // 清除脏数据标记
        cacheService.setWithExpire(cacheKey, userData, Duration.ofMinutes(30));
        return userData;
    }
    
    // 正常的缓存读取逻辑
    return getFromCacheOrDatabase(cacheKey, id);
}
```

## 扩展知识点

### 1. 分布式锁防止并发问题

```java
public void updateUserWithLock(Long id, UserData userData) {
    String lockKey = "lock:user:" + id;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        if (lock.tryLock(10, TimeUnit.SECONDS)) {
            // 执行更新操作
            updateUser(id, userData);
        }
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```
