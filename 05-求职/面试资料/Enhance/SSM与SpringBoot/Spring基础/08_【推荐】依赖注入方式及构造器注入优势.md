## 依赖注入的三种主要方式

### 1. 构造器注入 (Constructor Injection)

```java
@Service
public class UserService {
    private final UserRepository userRepository;
    private final EmailService emailService;
    
    // 构造器注入
    public UserService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }
}
```

### 2. Setter注入 (Setter Injection)

```java
@Service
public class UserService {
    private UserRepository userRepository;
    private EmailService emailService;
    
    @Autowired
    public void setUserRepository(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
    
    @Autowired
    public void setEmailService(EmailService emailService) {
        this.emailService = emailService;
    }
}
```

### 3. 字段注入 (Field Injection)

```java
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository;
    
    @Autowired
    private EmailService emailService;
}
```

## 构造器注入的优势

### 1. **保证依赖不可变性**

- 使用 `final` 关键字，确保依赖在对象创建后不会被修改
- 提高代码的线程安全性和可靠性

### 2. **强制依赖检查**

```java
// 如果缺少必要依赖，在应用启动时就会失败，而不是运行时
public UserService(UserRepository userRepository) {
    this.userRepository = Objects.requireNonNull(userRepository, "UserRepository不能为空");
}
```

### 3. **避免循环依赖**

Spring框架能在应用启动时检测构造器循环依赖并抛出异常，帮助开发者及早发现设计问题。

### 4. **更好的测试支持**

```java
@Test
public void testUserService() {
    // 不需要Spring容器，直接创建测试对象
    UserRepository mockRepo = Mockito.mock(UserRepository.class);
    EmailService mockEmail = Mockito.mock(EmailService.class);
    
    UserService userService = new UserService(mockRepo, mockEmail);
    // 进行测试...
}
```

### 5. **明确的依赖声明**

构造器参数清晰地展示了类的所有必需依赖，提高代码可读性。

## 相关扩展知识

### Spring Framework中的最佳实践

从Spring 4.3开始，如果类只有一个构造器，`@Autowired`注解可以省略：

```java
@Service
public class UserService {
    private final UserRepository userRepository;
    
    // 不需要@Autowired注解
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

### 依赖注入的设计原则

1. **依赖倒置原则 (DIP)**：高层模块不应依赖低层模块，都应依赖抽象
2. **控制反转 (IoC)**：对象创建和依赖关系由外部容器管理
3. **单一职责原则**：每个类只负责一个功能领域

### 实际应用场景对比

- **构造器注入**：适用于必需依赖，推荐作为默认选择
- **Setter注入**：适用于可选依赖或需要重新配置的场景
- **字段注入**：虽然简洁，但不推荐在生产代码中使用，主要用于测试

这种设计模式体现了现代Java开发中"约定优于配置"和"失败快速"的理念，有助于构建更加健壮和可维护的应用程序。
