# 三次握手与四次挥手

## 三次握手（Three-Way Handshake）

三次握手是TCP建立连接的过程，确保客户端和服务器都能正常收发数据。

### 握手过程：

1. **第一次握手（SYN）**
    
    - 客户端发送SYN包，序列号为x
    - 状态：客户端进入SYN_SENT状态
2. **第二次握手（SYN+ACK）**
    
    - 服务器收到SYN包，发送SYN+ACK包
    - 确认号为x+1，自己的序列号为y
    - 状态：服务器进入SYN_RCVD状态
3. **第三次握手（ACK）**
    
    - 客户端发送ACK包，确认号为y+1
    - 状态：双方都进入ESTABLISHED状态，连接建立成功

### 为什么需要三次握手？

- **防止失效连接请求**：避免因网络延迟导致的重复连接
- **确认双方收发能力**：确保双方都能正常发送和接收数据
- **同步序列号**：为后续数据传输建立正确的序列号

## 四次挥手（Four-Way Handshake）

四次挥手是TCP断开连接的过程，需要双方都确认连接关闭。

### 挥手过程：

1. **第一次挥手（FIN）**
    
    - 客户端发送FIN包，请求关闭连接
    - 状态：客户端进入FIN_WAIT_1状态
2. **第二次挥手（ACK）**
    
    - 服务器发送ACK包，确认收到关闭请求
    - 状态：客户端进入FIN_WAIT_2状态，服务器进入CLOSE_WAIT状态
3. **第三次挥手（FIN）**
    
    - 服务器发送FIN包，请求关闭连接
    - 状态：服务器进入LAST_ACK状态
4. **第四次挥手（ACK）**
    
    - 客户端发送ACK包，确认关闭
    - 状态：客户端进入TIME_WAIT状态，服务器进入CLOSED状态

### 为什么需要四次挥手？

- **全双工通信**：TCP是全双工的，双方都需要单独关闭
- **数据完整性**：确保所有数据都已传输完成
- **TIME_WAIT状态**：防止延迟的数据包影响新连接

## 相关扩展知识

### 常见异常情况：

- **半连接攻击（SYN Flood）**：大量SYN包占用服务器资源
- **TIME_WAIT过多**：可能导致端口耗尽
- **连接超时**：网络异常导致握手失败

### 优化策略：

- **SYN Cookie**：防御SYN Flood攻击
- **SO_REUSEADDR**：快速重用TIME_WAIT状态的端口
- **连接池**：减少频繁建立/关闭连接的开销

### 在Java中的应用：

```java
// Socket连接示例
Socket socket = new Socket("localhost", 8080); // 触发三次握手
// 数据传输...
socket.close(); // 触发四次挥手
```
