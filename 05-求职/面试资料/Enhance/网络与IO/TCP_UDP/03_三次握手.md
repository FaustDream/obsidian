# 三次握手

三次握手（Three-Way Handshake）是TCP协议建立连接的过程，确保客户端和服务器都能正常收发数据。让我详细解释这个过程：

## 三次握手的具体过程

**第一次握手（SYN）**

- 客户端向服务器发送SYN包（同步序列编号）
- 设置SYN=1，序列号seq=x（随机数）
- 客户端进入SYN_SENT状态

**第二次握手（SYN+ACK）**

- 服务器收到SYN包后，发送SYN+ACK包作为应答
- 设置SYN=1，ACK=1，序列号seq=y，确认号ack=x+1
- 服务器进入SYN_RCVD状态

**第三次握手（ACK）**

- 客户端收到SYN+ACK包后，发送ACK包确认
- 设置ACK=1，序列号seq=x+1，确认号ack=y+1
- 连接建立，双方进入ESTABLISHED状态

## 为什么需要三次握手？

**防止失效连接请求**

- 如果只有两次握手，可能出现这种情况：客户端发送的第一个连接请求在网络中滞留，等连接释放后才到达服务器
- 服务器误以为客户端要建立新连接，分配资源，但客户端实际上不会发送数据
- 三次握手确保客户端确实想要建立连接

**确认双方收发能力**

- 第一次握手：服务器确认客户端发送能力正常
- 第二次握手：客户端确认服务器收发能力正常
- 第三次握手：服务器确认客户端接收能力正常

## Java中的相关实现

```java
// 客户端Socket连接
Socket socket = new Socket("localhost", 8080);
// 这行代码执行时，底层会自动完成三次握手

// 服务器端
ServerSocket serverSocket = new ServerSocket(8080);
Socket clientSocket = serverSocket.accept(); // 等待客户端连接
```

## 扩展知识点

**四次挥手（连接释放）**

- TCP连接释放需要四次挥手，因为TCP是全双工协议
- 每个方向的连接都需要单独关闭

**半连接队列和全连接队列**

- 半连接队列：存储处于SYN_RCVD状态的连接
- 全连接队列：存储完成三次握手但还未被应用程序accept的连接

**SYN洪水攻击**

- 攻击者发送大量SYN包但不完成三次握手
- 导致服务器半连接队列满，无法处理正常请求
- 防护措施：SYN Cookie、限制半连接数量等

**Java NIO中的非阻塞连接**

```java
SocketChannel channel = SocketChannel.open();
channel.configureBlocking(false);
boolean connected = channel.connect(new InetSocketAddress("localhost", 8080));
if (!connected) {
    // 连接未完成，需要调用finishConnect()
    while (!channel.finishConnect()) {
        // 等待连接完成
    }
}
```

这些知识点在实际开发中都很重要，特别是在处理高并发网络应用时。
