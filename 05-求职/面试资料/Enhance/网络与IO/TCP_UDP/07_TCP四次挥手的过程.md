# TCP四次挥手的过程

TCP四次挥手是TCP连接终止的过程，让我详细解释一下：

## 四次挥手的具体过程

### 第一次挥手（FIN）

- **主动方**发送FIN报文段，表示"我没有数据要发送了"
- 此时主动方进入**FIN_WAIT_1**状态
- FIN报文段的序号为seq=u（等于前面已传送数据的最后一个字节的序号+1）

### 第二次挥手（ACK）

- **被动方**收到FIN后，发送ACK报文段进行确认
- 确认号ack=u+1，序号seq=v
- 被动方进入**CLOSE_WAIT**状态
- 主动方收到ACK后进入**FIN_WAIT_2**状态

### 第三次挥手（FIN）

- **被动方**发送完剩余数据后，发送FIN报文段
- 序号seq=w，确认号ack=u+1
- 被动方进入**LAST_ACK**状态

### 第四次挥手（ACK）

- **主动方**收到FIN后，发送最后的ACK报文段
- 确认号ack=w+1，序号seq=u+1
- 主动方进入**TIME_WAIT**状态，等待2MSL后关闭
- 被动方收到ACK后直接进入**CLOSED**状态

## 为什么需要四次挥手？

TCP是**全双工**通信，每个方向都需要单独关闭：

- 第一、二次挥手：关闭主动方→被动方的数据传输
- 第三、四次挥手：关闭被动方→主动方的数据传输

## TIME_WAIT状态的意义

主动关闭方需要等待2MSL（Maximum Segment Lifetime）：

1. **确保最后的ACK能够到达**：如果ACK丢失，被动方会重传FIN
2. **防止旧连接的数据包干扰新连接**

## 相关面试扩展点

### 1. 与三次握手的对比

- **握手**：建立连接，确认双方收发能力
- **挥手**：终止连接，确保数据完整传输

### 2. 异常情况处理

- **RST强制关闭**：不走四次挥手，直接重置连接
- **半关闭状态**：一方关闭发送，但仍可接收数据

### 3. 编程实践

```java
// Java中的socket关闭
socket.close(); // 会触发四次挥手过程
```

### 4. 性能优化

- **连接池**：避免频繁建立/关闭连接
- **长连接**：减少挥手开销
- **SO_LINGER**：控制close()行为

这个过程体现了TCP协议的可靠性设计，确保连接的优雅终止和数据的完整性。
