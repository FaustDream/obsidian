# HTTPS连接的建立过程？SSL/TLS握手？

## HTTPS连接建立过程

HTTPS连接的建立实际上是在TCP连接基础上增加了SSL/TLS安全层，整个过程可以分为两个阶段：

### 第一阶段：TCP三次握手

```
客户端 → 服务器：SYN
服务器 → 客户端：SYN-ACK  
客户端 → 服务器：ACK
```

### 第二阶段：SSL/TLS握手过程

**1. Client Hello（客户端问候）**

- 客户端发送支持的SSL/TLS版本
- 发送客户端随机数（Client Random）
- 发送支持的加密套件列表（如AES-256-GCM）
- 发送支持的压缩方法

**2. Server Hello（服务器问候）**

- 服务器选择SSL/TLS版本
- 发送服务器随机数（Server Random）
- 选择加密套件
- 发送服务器证书（包含公钥）

**3. 证书验证**

- 客户端验证服务器证书的有效性
- 检查证书是否过期、是否被信任的CA签发
- 验证证书中的域名是否匹配

**4. 密钥交换**

- 客户端生成预主密钥（Pre-Master Secret）
- 使用服务器公钥加密预主密钥发送给服务器
- 双方基于Client Random + Server Random + Pre-Master Secret生成主密钥

**5. 握手完成**

- 双方发送"Change Cipher Spec"消息
- 发送"Finished"消息（用新密钥加密）
- 握手完成，开始加密通信

## 相关扩展知识

### 对称加密 vs 非对称加密

- **握手阶段**：使用非对称加密（RSA/ECDHE）进行密钥交换
- **数据传输**：使用对称加密（AES）进行实际数据加密，效率更高

### 证书链验证

```
根证书CA → 中间证书CA → 服务器证书
```

### 现代优化技术

- **TLS 1.3**：简化握手过程，减少往返次数
- **HSTS**：强制使用HTTPS，防止降级攻击
- **Certificate Pinning**：固定证书，防止中间人攻击

### 性能考虑

- SSL/TLS握手相对耗时，可通过Session复用、连接复用来优化
- HTTP/2多路复用可以减少握手次数

这个过程确保了通信的**机密性**（加密）、**完整性**（防篡改）和**身份认证**（证书验证）。
