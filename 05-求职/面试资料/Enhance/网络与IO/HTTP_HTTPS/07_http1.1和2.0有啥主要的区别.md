# http1.1和2.0有啥主要的区别？

HTTP/1.1 和 HTTP/2.0 的主要区别可以从以下几个方面来理解：

## 1. 连接复用机制

**HTTP/1.1：**

- 默认使用持久连接（Keep-Alive），但同一个连接同时只能处理一个请求
- 需要等待前一个请求完成后才能发送下一个请求（队头阻塞问题）
- 浏览器通常会开启6-8个并行连接来缓解这个问题

**HTTP/2.0：**

- 引入多路复用（Multiplexing），一个连接可以同时处理多个请求和响应
- 将数据分割成帧（Frame），不同请求的帧可以交错发送
- 彻底解决了队头阻塞问题

## 2. 数据传输格式

**HTTP/1.1：**

- 基于文本协议，头部和体都是明文传输
- 头部信息冗余，每次请求都要重复发送相同的头部信息

**HTTP/2.0：**

- 基于二进制协议，传输效率更高
- 使用HPACK算法压缩头部，减少重复头部信息的传输
- 支持头部字段的索引表，进一步提高压缩效率

## 3. 服务器推送

**HTTP/1.1：**

- 只支持客户端主动请求，服务器被动响应
- 需要额外的技术（如WebSocket）来实现服务器主动推送

**HTTP/2.0：**

- 支持服务器推送（Server Push）
- 服务器可以主动向客户端推送资源，减少往返次数
- 比如请求HTML时，服务器可以主动推送CSS和JS文件

## 4. 流量控制

**HTTP/1.1：**

- 依赖TCP层的流量控制
- 无法对单个请求进行精细化的流量控制

**HTTP/2.0：**

- 在应用层实现了流量控制机制
- 可以对单个流（Stream）进行流量控制
- 支持流的优先级设置，重要资源可以优先传输

## 5. 性能对比

**HTTP/1.1的问题：**

- 队头阻塞导致延迟增加
- 头部冗余浪费带宽
- 需要多个连接增加服务器负担

**HTTP/2.0的优势：**

- 显著减少延迟，特别是在高延迟网络环境下
- 减少带宽消耗
- 更好的资源利用率

## 扩展知识

**HTTP/3.0的发展：** 虽然HTTP/2.0解决了应用层的队头阻塞，但TCP层仍然存在队头阻塞问题。HTTP/3.0基于QUIC协议，使用UDP传输，进一步解决了这个问题。

**实际应用考虑：**

- HTTP/2.0需要HTTPS支持
- 对于小文件多的场景，HTTP/2.0优势明显
- 需要考虑服务器和客户端的兼容性支持

这些区别直接影响了Web应用的性能和用户体验，在实际开发中需要根据具体场景选择合适的协议版本。
