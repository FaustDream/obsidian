# 单一职责原则怎么理解 举例子

单一职责原则（Single Responsibility Principle，SRP）是面向对象设计原则中的第一个原则，它的核心思想是：**一个类应该只有一个引起它变化的原因**。

## 核心理解

简单来说，一个类应该只负责一件事情，只有一个职责。如果一个类承担了多个职责，那么这些职责就耦合在一起，当其中一个职责需要变化时，可能会影响到其他职责的实现。

## 违反单一职责原则的例子

```java
// 违反单一职责原则的员工类
public class Employee {
    private String name;
    private String department;
    private double salary;
    
    // 员工基本信息管理
    public void setName(String name) {
        this.name = name;
    }
    
    public String getName() {
        return name;
    }
    
    // 薪资计算逻辑
    public double calculateSalary() {
        // 复杂的薪资计算逻辑
        return salary * 1.1;
    }
    
    // 数据库操作
    public void saveToDatabase() {
        // 数据库保存逻辑
        System.out.println("保存员工信息到数据库");
    }
    
    // 报表生成
    public void generateReport() {
        // 生成员工报表
        System.out.println("生成员工报表");
    }
}
```

**问题分析：** 这个类承担了4个职责：

1. 员工信息管理
2. 薪资计算
3. 数据库操作
4. 报表生成

当薪资计算规则改变时，需要修改这个类；当数据库结构改变时，也需要修改这个类；当报表格式改变时，还是需要修改这个类。

## 遵循单一职责原则的重构

```java
// 员工信息类 - 只负责员工基本信息
public class Employee {
    private String name;
    private String department;
    private double salary;
    
    public void setName(String name) {
        this.name = name;
    }
    
    public String getName() {
        return name;
    }
    
    public double getSalary() {
        return salary;
    }
    
    public void setSalary(double salary) {
        this.salary = salary;
    }
}

// 薪资计算类 - 只负责薪资计算
public class SalaryCalculator {
    public double calculateSalary(Employee employee) {
        // 复杂的薪资计算逻辑
        return employee.getSalary() * 1.1;
    }
}

// 数据库操作类 - 只负责数据持久化
public class EmployeeRepository {
    public void save(Employee employee) {
        // 数据库保存逻辑
        System.out.println("保存员工信息到数据库: " + employee.getName());
    }
    
    public Employee findById(Long id) {
        // 查询逻辑
        return new Employee();
    }
}

// 报表生成类 - 只负责报表生成
public class EmployeeReportGenerator {
    public void generateReport(Employee employee) {
        // 生成员工报表
        System.out.println("生成员工报表: " + employee.getName());
    }
}
```

## 实际应用场景

### 1. 用户管理系统

```java
// 用户信息
public class User {
    private String username;
    private String email;
    // getter/setter
}

// 用户验证
public class UserValidator {
    public boolean validateEmail(String email) {
        return email.contains("@");
    }
}

// 用户服务
public class UserService {
    private UserRepository userRepository;
    private UserValidator userValidator;
    
    public void createUser(User user) {
        if (userValidator.validateEmail(user.getEmail())) {
            userRepository.save(user);
        }
    }
}
```

### 2. 订单处理系统

```java
// 订单信息
public class Order {
    private String orderId;

    private double totalAmount;

public class OrderCalculator {

                .sum();
    }
}

public class OrderNotificationService {

    }
}
```

## 单一职责原则的好处

1. **降低类的复杂度**：每个类只负责一个职责，逻辑更清晰
2. **提高可读性和可维护性**：职责明确，代码更容易理解
3. **降低变更风险**：修改一个职责不会影响其他职责
4. **提高复用性**：单一职责的类更容易被其他模块复用

## 判断是否违反单一职责原则的技巧

1. **看类名**：如果类名中包含"And"、"Or"等连接词，可能违反了原则
2. **看方法**：如果类中的方法可以明显分为几组，每组处理不同的事情
3. **看变化原因**：如果有多个不同的原因需要修改这个类
4. **看依赖**：如果类依赖了很多不相关的外部类或包

单一职责原则是面向对象设计的基础，掌握好这个原则能让你写出更加清晰、可维护的代码。
