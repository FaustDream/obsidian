# 你觉得什么时候需要用到单例模式？

单例模式适用于以下几种典型场景：

## 核心使用场景

**1. 资源管理类**

- 数据库连接池：确保全局只有一个连接池实例，避免资源浪费
- 线程池：统一管理线程资源，防止重复创建
- 文件系统操作：确保文件访问的一致性

**2. 配置管理**

- 应用配置读取：全局配置信息只需要一个实例
- 系统参数管理：避免重复加载配置文件

**3. 日志记录**

- Logger实例：确保日志输出的统一性和性能
- 审计跟踪：统一的日志记录点

**4. 缓存管理**

- 内存缓存：Redis客户端、本地缓存等
- 会话管理：用户会话信息的统一管理

## 判断标准

什么时候考虑使用单例：

- 类的实例化成本很高（如数据库连接）
- 需要全局唯一的访问点
- 实例包含无状态的工具方法
- 需要统一管理某种资源

## 常见实现方式

```java
// 饿汉式（推荐）
public class DatabasePool {
    private static final DatabasePool INSTANCE = new DatabasePool();
    
    private DatabasePool() {}
    
    public static DatabasePool getInstance() {
        return INSTANCE;
    }
}

// 懒汉式（线程安全）
public class ConfigManager {
    private static volatile ConfigManager instance;
    
    private ConfigManager() {}
    
    public static ConfigManager getInstance() {
        if (instance == null) {
            synchronized (ConfigManager.class) {
                if (instance == null) {
                    instance = new ConfigManager();
                }
            }
        }
        return instance;
    }
}
```

## 注意事项

**何时不适用单例：**

- 需要多个实例的场景
- 有状态且需要并发访问的类
- 单元测试困难的情况
- 违反单一职责原则的复杂类

**Spring框架中的单例：** Spring默认bean都是单例的，但这是容器级别的单例，与传统单例模式有所不同。Spring的单例更灵活，支持作用域管理。

关键是要根据具体业务场景判断，不要为了使用设计模式而使用，要解决实际问题。
