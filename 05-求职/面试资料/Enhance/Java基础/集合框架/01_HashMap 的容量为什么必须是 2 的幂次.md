Java HashMap 的容量必须是 2 的幂次，这是一个精心设计的决策，主要原因如下：

## 1. 高效的取模运算

当容量是 2 的幂次时，可以用位运算替代取模运算：

```java
// 传统取模运算（较慢）
int index = hash % capacity;

// 位运算优化（更快）
int index = hash & (capacity - 1);
```

例如，当容量为 16 时：

- 16 的二进制：`10000`
- 15 的二进制：`01111`
- `hash & 15` 等价于 `hash % 16`，但位运算比取模运算快得多

## 2. 均匀的哈希分布

2 的幂次容量能确保哈希值在桶中的分布更加均匀。如果容量不是 2 的幂次，某些桶可能会比其他桶接收到更多的元素，导致哈希冲突增加。

## 3. 扩容时的高效重新分布

HashMap 扩容时容量会翻倍（从 2^n 变为 2^(n+1)）。在重新计算元素位置时：

- 原来在位置 `i` 的元素，扩容后要么还在位置 `i`
- 要么移动到位置 `i + oldCapacity`

这种规律性使得扩容过程更加高效，无需重新计算所有元素的哈希值。

## 4. 源码实现

在 HashMap 的源码中可以看到相关实现：

```java
static final int tableSizeFor(int cap) {
    int n = cap - 1;
    n |= n >>> 1;
    n |= n >>> 2;
    n |= n >>> 4;
    n |= n >>> 8;
    n |= n >>> 16;
    return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;
}
```

这个方法确保无论输入什么初始容量，最终都会被调整为不小于该值的最小 2 的幂次。

## 总结

2 的幂次容量设计是性能优化的重要体现，它通过位运算加速、均匀分布和高效扩容等机制，显著提升了 HashMap 的整体性能。这也是为什么即使你指定了其他初始容量值，HashMap 也会自动调整为最接近的 2 的幂次的原因。

# 疑问：为什么 `hash & 15` 等价于 `hash % 16`？

这是一个很好的问题！让我详细解释为什么 `hash & 15` 等价于 `hash % 16`。

## 数学原理

当除数是 2 的幂次时，取模运算可以用位运算来替代。关键在于理解二进制的特性：

**16 的二进制表示：**

- 16 = 2^4 = `10000`（二进制）
- 15 = 16-1 = `01111`（二进制）

## 具体例子

让我们用几个具体的哈希值来验证：

### 例子 1：hash = 25

```
25 的二进制：11001
16 的二进制：10000
15 的二进制：01111

取模运算：25 % 16 = 9
位运算：  25 & 15 = 11001 & 01111 = 01001 = 9
```

### 例子 2：hash = 42

```
42 的二进制：101010
15 的二进制：001111

取模运算：42 % 16 = 10
位运算：  42 & 15 = 101010 & 001111 = 001010 = 10
```

### 例子 3：hash = 100

```
100 的二进制：1100100
15 的二进制： 0001111

取模运算：100 % 16 = 4
位运算：  100 & 15 = 1100100 & 0001111 = 0000100 = 4
```

## 原理解释

**为什么会相等？**

当我们对一个数进行 `% 16` 运算时，实际上是要找到这个数除以 16 的余数。在二进制中：

- 16 = `10000`，意味着第 5 位（从右数第 5 位）是 1，其他低位都是 0
- 对 16 取模，等价于只保留低 4 位的值
- 15 = `01111`，恰好是低 4 位全为 1 的掩码

**位运算的作用：**

```
任意数字：  xxxx xxxx xxxx 1011
与15求AND： 0000 0000 0000 1111
结果：      0000 0000 0000 1011
```

`& 15` 运算会将高位全部清零，只保留最低的 4 位，这正好等价于对 16 取模的结果。

## 通用规律

这个规律适用于所有 2 的幂次：

- `hash % 2` = `hash & 1`
- `hash % 4` = `hash & 3`
- `hash % 8` = `hash & 7`
- `hash % 16` = `hash & 15`
- `hash % 32` = `hash & 31`

一般来说：**当 n = 2^k 时，`hash % n` = `hash & (n-1)`**

## 性能优势

位运算比取模运算快得多，因为：

- 位运算是 CPU 的基本操作，只需要 1 个时钟周期
- 取模运算需要除法，可能需要多个时钟周期
- 在高频操作的 HashMap 中，这种优化会带来显著的性能提升

这就是为什么 HashMap 坚持使用 2 的幂次作为容量的核心原因——既保证了正确性，又获得了性能优势。
