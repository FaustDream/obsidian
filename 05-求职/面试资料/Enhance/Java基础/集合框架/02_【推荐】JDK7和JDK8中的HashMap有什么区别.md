# JDK7和JDK8中的HashMap有什么区别

JDK7和JDK8中的HashMap主要有以下几个重要区别：

## 1. 底层数据结构的变化

**JDK7**: 使用 **数组 + 链表** 的结构

- 当发生哈希冲突时，新元素直接插入到链表头部（头插法）

**JDK8**: 使用 **数组 + 链表 + 红黑树** 的结构

- 当链表长度超过8且数组长度≥64时，链表会转换为红黑树
- 当红黑树节点数量少于6时，会退化回链表

## 2. 插入方式的改变

**JDK7**: 头插法

```java
// 新节点插入到链表头部
newNode.next = table[index];
table[index] = newNode;
```

**JDK8**: 尾插法

```java
// 新节点插入到链表尾部
lastNode.next = newNode;
```

## 3. 扩容机制的优化

**JDK7**:

- 扩容时需要重新计算每个元素的hash值
- 在多线程环境下可能出现环形链表问题

**JDK8**:

- 利用位运算优化，元素要么在原位置，要么在"原位置+旧容量"的位置
- 解决了环形链表问题

## 4. 性能提升

**查找效率**:

- JDK7: 链表查找时间复杂度 O(n)
- JDK8: 红黑树查找时间复杂度 O(log n)

**扩容效率**:

- JDK8的扩容算法更加高效，减少了rehash的计算量

## 5. 相关的扩展知识

### 为什么选择8作为树化阈值？

这是基于泊松分布的统计结果，链表长度达到8的概率非常低（约0.00000006），这样既避免了频繁的树化操作，又保证了性能。

### 线程安全问题

两个版本的HashMap都不是线程安全的，但JDK8修复了JDK7中多线程扩容时可能出现的死循环问题。

### 替代方案

- 线程安全：使用==ConcurrentHashMap==
- 插入顺序：使用LinkedHashMap
- 排序需求：使用TreeMap

这些改进使得JDK8的HashMap在处理大量数据和高并发场景下表现更加出色。
