# HashMap是如何解决哈希冲突的

HashMap解决哈希冲突主要采用了两种核心机制：**链地址法（拉链法）** 和 **红黑树优化**。让我详细解释这个过程：

## 1. 链地址法（拉链法）- 主要解决方案

当多个key经过hash函数计算后得到相同的数组索引时，HashMap会在该位置形成一个链表：

```java
// 简化的HashMap结构
class Node<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;  // 指向下一个节点
}
```

**工作原理：**

- 相同hash值的元素会以链表形式存储在同一个桶中
- 新元素默认插入到链表头部（JDK 1.8之前是头插法，1.8改为尾插法）
- 查找时遍历链表，通过equals()方法比较key值

## 2. 红黑树优化（JDK 1.8引入）

当链表长度超过阈值时，会转换为红黑树以提高查找效率：

**转换条件：**

- 链表长度 ≥ 8 且 数组长度 ≥ 64：链表转红黑树
- 红黑树节点数 ≤ 6：红黑树转回链表

**性能对比：**

- 链表查找：O(n)
- 红黑树查找：O(log n)

## 3. 扩容机制

当负载因子超过0.75时，HashMap会进行扩容：

```java
// 扩容过程
void resize() {
    // 1. 创建新数组（容量翻倍）
    Node<K,V>[] newTab = new Node[oldCap << 1];
    
    // 2. 重新计算每个元素的位置
    for (int j = 0; j < oldTab.length; ++j) {
        Node<K,V> e = oldTab[j];
        if (e != null) {
            // 重新hash并分配到新位置
        }
    }
}
```

## 4. hash函数优化

HashMap使用了扰动函数来减少哈希冲突：

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

这个设计让高位和低位都参与到最终的hash计算中，减少了冲突概率。

## 扩展知识点

**为什么选择8作为链表转红黑树的阈值？**

- 基于泊松分布的统计学计算
- 链表长度达到8的概率约为0.00000006
- 平衡了时间和空间复杂度

**JDK版本演进：**

- JDK 1.7：数组 + 链表
- JDK 1.8：数组 + 链表 + 红黑树
- 插入方式从头插法改为尾插法（避免并发环境下的死循环）

这种设计使HashMap在大多数情况下都能保持O(1)的时间复杂度，即使在最坏情况下也能维持O(log n)的性能。
