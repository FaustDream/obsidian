HashMap扩容采用2倍机制和0.75扩容因子的设计有深层的技术原因：

## 为什么扩容是2倍？

**1. 位运算优化**

- HashMap使用 `hash & (length-1)` 来计算数组索引
- 当容量是2的幂次时，`length-1` 的二进制全是1（如16-1=15，二进制1111）
- 这样位运算等价于取模运算，但性能更高

**2. 重新散列的高效性** 
扩容时，元素要么保持原位置，要么移动到 `原位置 + 老容量` 的位置。这是因为：

- 老容量下：`hash & (oldCap-1)`
- 新容量下：`hash & (newCap-1)`
- 由于新容量是老容量的2倍，只需要看hash值在老容量对应位上是0还是1

```java
// 举例：容量从8扩到16
// hash值为5（二进制101）
// 老容量：5 & (8-1) = 5 & 7 = 101 & 111 = 5
// 新容量：5 & (16-1) = 5 & 15 = 101 & 1111 = 5
// 位置不变

// hash值为13（二进制1101）
// 老容量：13 & 7 = 1101 & 111 = 5
// 新容量：13 & 15 = 1101 & 1111 = 13
// 移动到5+8=13位置
```

## 为什么扩容因子是0.75？

**1. 时间与空间的平衡**

- 太小（如0.5）：空间浪费严重，但冲突少
- 太大（如0.9）：空间利用率高，但冲突频繁
- 0.75在两者间取得较好平衡

**2. 数学依据** 根据泊松分布，当负载因子为0.75时：

- 空间利用率较高
- 冲突概率相对较低
- 查找性能较好

**3. 实际测试验证** Java开发团队通过大量测试发现，0.75是一个经验最优值，在实际应用中表现最佳。

这种设计使得HashMap在保持高性能的同时，合理利用内存空间，是经过深思熟虑的工程实践结果。
