# 前端需求表达术语与模板

你缺的不是表达能力，是缺一套前端提需求时可复用的固定句式和标准术语。

## 万能表达公式

`位置 + 对象 + 当前问题 + 触发条件 + 目标效果 + 验收标准`

示例：

`订单页顶部筛选区的日期选择器，在 1280px 以下会换行，导致查询按钮掉到第二行。改成同一行右对齐；空间不够时把次要筛选项收进“更多筛选”；所有控件高度统一 32px。`

## 效果图与设计阶段术语

- `线框图` = 只看布局结构，不看颜色和细节
- `原型图` = 看页面流程和交互
- `高保真稿` = 接近最终效果的设计图
- `交互稿` = 说明点击、切换、悬停、校验等行为
- `设计稿` = 视觉样式整体叫法

## 口语转术语对照

- “上面那一排按钮” -> `工具栏` / `操作区`
- “左边菜单” -> `侧边导航`
- “弹出来的小框” -> `弹窗` / `抽屉` / `popover`
- “那个提示字” -> `placeholder` / `tooltip` / `toast`
- “下面翻页那个” -> `分页器`
- “中间空白提示” -> `空状态`
- “一格一格的数据” -> `表格` / `卡片列表`
- “切来切去的标签” -> `tabs`

## 给 AI 提需求的标准句式

`在 [页面/模块] 的 [组件/区域]，现在存在 [问题]；在 [条件] 下会出现 [现象]；请改成 [目标效果]；要求 [尺寸/对齐/状态/交互/兼容规则]。`

示例：

`在用户管理页的表格操作列，按钮文字过长时会把整行撑高；在 150% 浏览器缩放下更明显。请改成固定高度并超出省略，鼠标悬停显示完整文本；要求操作列垂直居中，对齐其他列。`

## 使用原则

- 不要说“这里有点怪”“这里改一下”
- 要说“哪个区域 + 什么组件 + 什么状态 + 什么问题 + 改成什么”
- 需求越具体，AI 和前端越容易一次理解
