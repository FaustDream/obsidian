# 自定义代码开发指导规范

以下是优化后的文本：

1. **禁止使用过时的类或方法（[Obsolete]标识）**
说明：C#中标记为过时的方法可能会在新版本的.NET Framework或官方接口中被移除，因此必须避免使用这些方法或类。

1. **避免调用******`Object.Equals`******方法导致空引用异常**
推荐使用常量或确保对象已初始化后调用`Equals`。
反例：
`user.Equals("Jack");`
推荐写法：
`"Jack".Equals(user);`

1. **避免在循环中调用Engine接口**
请尽量减少服务间的通信次数：跨进程接口调用的耗时和CPU消耗大约是普通接口调用的10,000倍。如果需要多次调用，建议使用批量接口。
示例：

```csharp

H3.DataModel.BulkCommit commit = new H3.DataModel.BulkCommit();
H3.DataModel.BizObjectSchema schema = this.Request.Engine.BizObjectManager.GetPublishedSchema("D0001001");
for (int i = 0; i < 1000; i++) {
    H3.DataModel.BizObject obj = new H3.DataModel.BizObject(this.Request.Engine, schema, this.Request.UserContext.UserId);
    obj["F000001"] = 1;
    obj.Create(commit);  // 如果对象已存在，可以使用Update
}
string errorMsg = null;
commit.Commit(this.Request.Engine.BizObjectManager, out errorMsg);
```

1. **禁止使用反射**
`GetType().Get*()`方法只能在单线程环境下执行。多线程环境中使用这些方法会降低性能。反射的执行速度约为普通代码的10倍。

1. **禁止对数据库进行DDL操作**
数据库的DDL操作，如创建（CREATE）、删除（DROP）和修改（ALTER）数据库和表结构，禁止在应用程序中直接操作。

1. **禁止对数据库进行大数据量操作**
避免一次性读取过多数据，如一次性读取上千条记录。

1. **禁止循环调用数据库操作**
避免在循环中频繁调用数据库操作，影响性能。

1. **避免通过SQL直接修改数据库数据**
不要通过SQL语句直接对业务数据进行增删改操作，应该通过接口进行操作。

1. **前端代码禁用循环调用服务端接口**
前端应该避免循环调用服务端接口，应构造一次性调用，返回统一处理结果。

1. **前端子表行添加时，使用批量添加**
在往子表填充数据时，尽量采用批量添加的方式。

1. **建议赋值时使用对象而非仅赋值ID**
在赋值人员、部门或关联数据时，尽量使用对象而非仅赋值ID。

1. **乐观锁机制**
为了提高系统性能并减少锁冲突，氚云中采用了乐观锁机制。
遇到以下提示报错时，优先考虑乐观锁问题：

 常见的易错场景：

 建议：在提交时仅更新数据对象，避免多次提交操作。如果需要多次更新，请在每次更新前重新加载数据并提交更新。

  - `保存失败: UpdateFailed`

  - `保存失败: ItemExists`

  - `Optimistic lock failed, the object has been updated by other user`

  1. 在`base.OnSubmit`前调用`this.Request.BizObject.Update()`；

  1. 在`base.OnSubmit`前调用`this.Request.Engine.WorkflowInstanceManager.SendMessage`激活当前流程；

  1. 在`base.OnSubmit`前调用自定义接口更新当前数据；

  1. 在`base.OnSubmit`前更新或新增子表数据。
